## 一、理论基础

### 1. MySQL架构

```
┌────────────────────────────────────────────────────────────┐
│                        MySQL Server                          │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                    Connection Layer                    │ │
│  │   连接管理器 → 线程池 → 用户认证 → 命令分发            │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                     SQL Layer                          │ │
│  │   Parser → Preprocessor → Optimizer → Executor        │ │
│  │   查询缓存（8.0移除）                                  │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                  Storage Engine Layer                   │ │
│  │         InnoDB    MyISAM    Memory    其他            │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐ │
│  │                     File Layer                         │ │
│  │   表空间(.ibd)  日志文件(.redo)  系统表空间             │ │
│  └────────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│                    OS & Disk                                │
└────────────────────────────────────────────────────────────┘
```

### 2. InnoDB架构

```
┌─────────────────────────────────────────────────────────────┐
│                        InnoDB架构                            │
│                                                              │
│  ┌───────────────┐        ┌───────────────────────────────┐ │
│  │   Buffer Pool │◄──────►│     Change Buffer              │ │
│  │   (内存池)     │        │  (写缓存，减少随机IO)          │ │
│  └───────┬───────┘        └───────────────────────────────┘ │
│          │                                                      │
│  ┌───────▼───────┐        ┌───────────────────────────────┐ │
│  │ Adaptive Hash │        │      Log Buffer                │ │
│  │   Index (AHI) │        │      (日志缓冲)                │ │
│  └───────────────┘        └───────────────┬───────────────┘ │
│                                            │                   │
│  ┌─────────────────────────────────────────▼───────────────┐ │
│  │              Doublewrite Buffer（双写缓冲）              │ │
│  │              防止部分页写入失败                          │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                    系统表空间                             │ │
│  │    data dictionary / undo segment / insert buffer       │ │
│  └─────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                  独立表空间(.ibd)                         │ │
│  │    每表一个表空间，存储数据和索引                        │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 3. InnoDB关键特性

```
事务支持：
  - ACID特性（原子性、一致性、隔离性、持久性）
  - MVCC（多版本并发控制）
  - 锁机制（行锁、表锁、意向锁）

索引结构：
  - B+树索引（聚簇索引 + 辅助索引）
  - 自适应哈希索引（AHI）
  - 全文索引（5.6+）

崩溃恢复：
  - Redo Log（重做日志）
  - Undo Log（回滚日志）
  - Doublewrite Buffer（双写缓冲）

内存管理：
  - Buffer Pool（缓冲池）
  - Change Buffer（变更缓冲）
  - Lock Info（锁信息）
```

### 4. 存储引擎对比

| 特性 | InnoDB | MyISAM | Memory |
|------|--------|--------|--------|
| 事务 | 支持 | 不支持 | 不支持 |
| 外键 | 支持 | 不支持 | 不支持 |
| 行锁 | 支持 | 表锁 | 支持 |
| 全文索引 | 5.6+支持 | 原生支持 | 不支持 |
| 存储限制 | 64TB | 256TB | RAM |
| 崩溃恢复 | 自动 | 需要修复 | 无 |
| 适用场景 | 通用/OLTP | 只读/OLAP | 临时表/缓存 |

## 二、实践应用

### 1. 核心配置

```ini
# my.cnf 核心配置
[mysqld]
# 基础设置
server-id = 1
port = 3306
datadir = /var/lib/mysql
socket = /var/lib/mysql/mysql.sock

# Buffer Pool配置（最重要！）
innodb_buffer_pool_size = 12G  # 通常为物理内存70-80%
innodb_buffer_pool_instances = 8  # 分成多个实例，减少竞争
innodb_buffer_pool_load_at_startup = ON  # 预热
innodb_buffer_pool_dump_at_shutdown = ON  # 保存热数据

# 日志配置
innodb_log_file_size = 2G  # Redo日志大小
innodb_log_buffer_size = 64M
innodb_flush_log_at_trx_commit = 1  # 1=强持久化，2=每秒刷盘，0=最快但可能丢数据

# IO配置
innodb_flush_method = O_DIRECT  # 绕过OS缓存
innodb_io_capacity = 2000  # SSD可提高
innodb_io_capacity_max = 4000
innodb_read_io_threads = 8
innodb_write_io_threads = 8

# 表空间
innodb_file_per_table = ON  # 独立表空间

# 连接数
max_connections = 2000
wait_timeout = 600
```

### 2. 状态监控

```sql
-- 查看Buffer Pool状态
SHOW ENGINE INNODB STATUS\G
-- 关键信息：
-- Total memory allocated = xxx
-- Buffer pool size = xxx pages

-- 查看连接状态

```

### 3. 性能分析

```sql
-- 查看当前查询
SHOW PROCESSLIST;
-- 或
SELECT * FROM information_schema.PROCESSLIST 
WHERE COMMAND != 'Sleep' ORDER BY TIME DESC;

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
SELECT * FROM information_schema.INNODB_LOCKS;

-- 查看事务
SELECT * FROM information_schema.INNODB_TRX;

-- 分析查询
EXPLAIN SELECT * FROM users WHERE id = 1\G
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 1\G  -- 8.0
```

## 三、生产环境问题案例

### 案例1：Buffer Pool设置不当导致性能差

**问题现象：**
MySQL查询很慢，CPU不高，IO高。

**分析过程：**
```sql
-- 1. 查看Buffer Pool状态
SHOW ENGINE INNODB STATUS\G
-- 发现：Buffer pool size = 134217728 pages (约500GB)
-- 但服务器只有64GB内存！

-- 2. 查看实际配置
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';

-- 3. 分析
-- Buffer Pool > 物理内存 = 严重swap
```

**根因分析：**
Buffer Pool设置超过物理内存，导致大量使用Swap，性能骤降。

**解决方案：**
```ini
[mysqld]
# 正确配置：物理内存的70-80%
innodb_buffer_pool_size = 48G
# 不要超过物理内存！
```

**经验教训：**
- Buffer Pool不能超过物理内存
- 预留OS和其他应用内存
- 监控Swap使用

### 案例2：长事务阻塞

**问题现象：**
部分查询长时间不返回，连接堆积。

**分析过程：**
```sql
-- 1. 查看长事务
SELECT * FROM information_schema.INNODB_TRX 
WHERE trx_started < NOW() - INTERVAL 10 MINUTE;

-- 2. 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 3. 查看具体锁
SELECT * FROM information_schema.INNODB_LOCKS;

-- 4. 分析
-- 找到trx_mysql_thread_id
-- 确认是哪个应用/连接
```

**根因分析：**
大事务长时间持有锁，后续请求被阻塞。

**解决方案：**
```sql
-- 1. 杀死长事务（紧急）
KILL <thread_id>;

-- 2. 优化事务：
-- - 拆分大事务为小事务
-- - 避免在事务中执行不必要的查询
-- - 调整事务隔离级别

-- 3. 设置超时
SET GLOBAL innodb_lock_wait_timeout = 5;  -- 默认50秒
SET GLOBAL max_execution_time = 30000;   -- 8.0 查询超时
```

**经验教训：**
- 监控长事务
- 应用层避免长事务
- 设置合理的超时时间

### 案例3：ibdata1文件过大

**问题现象：**
系统表空间文件ibdata1很大，磁盘空间不足。

**分析过程：**
```bash
# 1. 查看文件大小
ls -lh /var/lib/mysql/ibdata1
# 500GB！

# 2. 查看表空间使用
SELECT table_schema, SUM(data_length + index_length) / 1024 / 1024 AS MB
FROM information_schema.tables
GROUP BY table_schema
ORDER BY MB DESC;
```

**根因分析：**
使用了共享表空间（innodb_file_per_table=OFF），Undo Log也在系统表空间。

**解决方案：**
```sql
-- 方案1：清理Undo Log
SET GLOBAL innodb_purge_rseg_truncate_frequency = 1;
SET GLOBAL innodb_undo_log_truncate = ON;

-- 方案2：重建数据库（彻底）
-- 1. mysqldump导出所有数据
-- 2. 停止MySQL
-- 3. 删除ibdata*和undo文件
-- 4. 配置innodb_file_per_table = ON
-- 5. 启动MySQL，导入数据
```

**经验教训：**
- 新部署必须设置innodb_file_per_table=ON
- 定期清理Undo Log
- 监控表空间增长
