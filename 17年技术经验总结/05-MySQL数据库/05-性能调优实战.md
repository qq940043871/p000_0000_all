## 一、慢查询分析

### 1. 开启慢查询日志

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;  -- 超过1秒记录
SET GLOBAL log_queries_not_using_indexes = 'ON';

-- 查看配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
```

### 2. 分析慢查询

```sql
-- 使用EXPLAIN
EXPLAIN SELECT * FROM orders WHERE status = 1\G

-- EXPLAIN ANALYZE（8.0+）
EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 1\G

-- 查看执行计划关键：
-- type: ALL(全表) < index < range < ref < eq_ref < const
-- key: 使用的索引
-- rows: 预计扫描行数
-- Extra: Using filesort(需排序), Using index(覆盖索引), Using temporary(临时表)
```

### 3. pt-query-digest

```bash
# 安装
yum install -y percona-toolkit

# 分析慢查询
pt-query-digest /var/log/mysql/slow.log

# 输出：
-- 统计TOP查询
-- 每个查询的：
--   - 执行次数
--   - 平均时间
--   - 总时间
--   - 执行计划
```

## 二、配置优化

### 1. 内存相关

```ini
[mysqld]
# Buffer Pool（最重要）
innodb_buffer_pool_size = 12G  # 物理内存70-80%

# 排序缓存
sort_buffer_size = 4M
join_buffer_size = 4M

# 读缓存
read_buffer_size = 2M
read_rnd_buffer_size = 8M
```

### 2. IO相关

```ini
[mysqld]
# Redo日志
innodb_log_file_size = 2G
innodb_log_buffer_size = 64M
innodb_flush_log_at_trx_commit = 1  # 数据安全优先

# IO线程
innodb_io_capacity = 2000
innodb_io_capacity_max = 4000
innodb_read_io_threads = 8
innodb_write_io_threads = 8
```

### 3. 连接相关

```ini
[mysqld]
# 最大连接
max_connections = 2000

# 连接超时
wait_timeout = 600
interactive_timeout = 600

# 线程缓存
thread_cache_size = 64

# 表缓存
table_open_cache = 4000
table_definition_cache = 2000
```

## 三、SQL优化

### 1. 常见优化

```sql
-- 1. 避免SELECT *
SELECT id, name, email FROM users WHERE id = 1;

-- 2. LIMIT优化
-- 深度分页优化
-- 旧：SELECT * FROM orders LIMIT 10000, 10
-- 新：
SELECT * FROM orders 
WHERE id > (SELECT id FROM orders ORDER BY id LIMIT 10000, 1)
ORDER BY id
LIMIT 10;

-- 3. 批量插入
INSERT INTO users VALUES (1,'A'),(2,'B'),(3,'C');
-- 比逐条INSERT快10倍

-- 4. 预编译SQL
PREPARE stmt FROM 'SELECT * FROM users WHERE id = ?';
SET @id = 1;
EXECUTE stmt USING @id;

-- 5. 临时表使用
-- 避免在WHERE中对字段使用函数
-- 旧：WHERE YEAR(create_time) = 2024
-- 新：WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'
```

### 2. 索引优化

```sql
-- 1. 合理使用复合索引
-- 索引INDEX(a,b,c)
-- 适合：a=? 或 a=? AND b=? 或 a=? AND b=? AND c=?

-- 2. 覆盖索引
-- 避免回表
-- SELECT id, name FROM users WHERE name = '张三'
-- 索引INDEX(name, id) 覆盖查询

-- 3. 前缀索引
ALTER TABLE orders ADD KEY (order_no(20));

-- 4. 索引下推（ICP）
-- 5.6+自动优化
-- 减少回表次数
```

## 四、生产问题案例

### 案例1：Buffer Pool命中率低

**问题现象：**
频繁磁盘IO，查询缓慢。

**分析过程：**
```sql
-- 查看Buffer Pool状态
SHOW ENGINE INNODB STATUS\G
-- Database pages: xxx
-- Free pages: xxx
-- Buffer pool hit rate: 950/1000 (95%)
-- 发现命中率偏低

-- 查看脏页
SHOW ENGINE INNODB STATUS\G
-- Modified db pages: xxx
-- 如果过多，刷盘影响性能
```

**解决方案：**
```ini
# 1. 预热Buffer Pool
innodb_buffer_pool_load_at_startup = ON
innodb_buffer_pool_dump_at_shutdown = ON

# 2. 分成多个实例
innodb_buffer_pool_instances = 8

# 3. 监控并优化
SHOW ENGINE INNODB STATUS;
```

### 案例2：连接数打满

**问题现象：**
应用报错"Too many connections"。

**分析过程：**
```sql
-- 查看连接
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';

-- 查看所有连接
SHOW PROCESSLIST;
-- 找到异常连接：
-- - Sleep状态过多
-- - 长时间运行查询
```

**解决方案：**
```sql
-- 1. 杀掉空闲连接
SELECT CONCAT('KILL ', id, ';') 
FROM information_schema.processlist 
WHERE Command = 'Sleep' AND Time > 3600;

-- 2. 调整最大连接
SET GLOBAL max_connections = 3000;

-- 3. 应用优化
-- - 使用连接池
-- - 减少长查询
-- - 设置合理的wait_timeout
```
