## 一、理论基础

### 1. 主从复制原理

```
主从复制架构：

┌──────────────┐         ┌──────────────┐
│   Master     │  ───►   │    Slave     │
│  (主库)      │ binlog   │   (从库)     │
│              │ ───►    │              │
│  ┌────────┐ │ relaylog │  ┌────────┐ │
│  │ 用户写入│ │         │  │ SQL线程│ │
│  └────┬───┘ │         │  └──┬───┘ │
│       │     │         │     │     │
│  ┌────▼───┐ │         │ ┌──▼────┐ │
│  │ binlog  │───────────►│relaylog│ │
│  └────────┘ │         │ └────────┘ │
└──────────────┘         └──────────────┘

复制流程：
  1. Master执行SQL，产生binlog
  2. Masterdump线程发送binlog到Slave
  3. Slave的IO线程接收，写入relaylog
  4. Slave的SQL线程回放relaylog
  5. 数据同步完成
```

### 2. 复制方式

```
异步复制（默认）：
  - Master执行后立即返回
  - Slave异步同步
  - 可能存在延迟

半同步复制：
  - Master等待至少一个Slave确认
  - 保证至少一个Slave收到日志
  - 性能略有下降

全同步复制：
  - Master等待所有Slave确认
  - 性能差，很少使用

GTID复制（5.7+推荐）：
  - 事务ID全局唯一
  - 自动定位位置
  - 故障切换更简单
```

### 3. 常见架构

```
┌─────────┐
│ VIP/LVS │  ← 入口
└────┬────┘
     │
┌────▼────┐
│ HAProxy │  ← 负载均衡
└────┬────┘
     │
 ┌───┴───┐
 │       │
┌▼──┐ ┌──▼┐
│ M1 │ │ M2 │  ← 双主（互相同步）
└┬──┘ └──┬─┘
 │        │
 │  ┌────┴────┐
 │  │         │
└┬─▼──┐ ┌──▼─┴┐
│ S1  │ │ S2  │  ← 从库
└─────┘ └─────┘

MHA架构：
  - Master High Availability
  - 自动监控主库
  - 自动选主和故障转移
```

## 二、实践应用

### 1. 主从配置

```ini
# Master配置
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW  # 推荐
sync_binlog = 1       # 强同步
gtid_mode = ON
enforce_gtid_consistency = ON

# 从库配置
[mysqld]
server-id = 2
relay-log = relay-bin
read_only = ON        # 从库只读
super_read_only = ON   # 禁止SUPER权限写入
gtid_mode = ON
enforce_gtid_consistency = ON
log_slave_updates = ON
```

```sql
-- Master创建复制账号
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- Slave配置复制
CHANGE MASTER TO
  MASTER_HOST = 'master_ip',
  MASTER_USER = 'repl',
  MASTER_PASSWORD = 'password',
  MASTER_AUTO_POSITION = 1;  -- GTID模式

-- 启动复制
START SLAVE;

-- 查看状态
SHOW SLAVE STATUS\G
-- 关键指标：
-- Slave_IO_Running: IO线程状态
-- Slave_SQL_Running: SQL线程状态
-- Seconds_Behind_Master: 延迟（秒）
-- Retrieved_Gtid_Set: 已接收的GTID
-- Executed_Gtid_Set: 已执行的GTID
```

### 2. 复制监控

```sql
-- 查看复制状态
SHOW SLAVE STATUS\G

-- 查看延迟
SHOW SLAVE STATUS\G | grep Seconds_Behind

-- 查看详细状态
SHOW PROCESSLIST;

-- 监控主库
SHOW MASTER STATUS;
SHOW BINARY LOGS;
```

### 3. 故障处理

```sql
-- 1. 主库故障切换
STOP SLAVE;
RESET SLAVE ALL;
-- 提升从库为主库
SET GLOBAL read_only = OFF;
-- 修改应用连接新的主库

-- 2. 从库数据修复
-- 方法1：从主库重新同步
STOP SLAVE;
RESET SLAVE ALL;
CHANGE MASTER TO ...
START SLAVE;

-- 方法2：使用pt-table-checksum检查一致性
-- 方法3：pt-table-sync修复差异

-- 3. GTID跳过一个事务
SET GTID_NEXT = 'uuid:transaction_num';
BEGIN; COMMIT;
SET GTID_NEXT = AUTOMATIC;
```

## 三、生产环境问题案例

### 案例1：主从延迟

**问题现象：**
从库数据明显落后主库，用户读取到旧数据。

**分析过程：**
```sql
-- 1. 查看延迟
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master: 3600  ← 延迟1小时！

-- 2. 分析原因
-- 查看Slave状态
SHOW PROCESSLIST;
-- SQL线程：处理慢（大量复杂查询）
-- IO线程：接收正常

-- 3. 查看从库负载
SHOW STATUS LIKE 'Threads%';
SHOW PROCESSLIST;

-- 4. 常见原因：
-- - 从库硬件差
-- - 大事务在从库执行
-- - 从库负载高
-- - 网络延迟
```

**解决方案：**
1. 从库配置优化
   ```ini
   innodb_flush_log_at_trx_commit = 2
   sync_binlog = 0
   ```
2. 读写分离优化
3. 使用并行复制
   ```sql
   STOP SLAVE;
   SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
   SET GLOBAL slave_parallel_workers = 8;
   START SLAVE;
   ```
4. 减少大事务

### 案例2：主库故障数据丢失

**问题现象：**
主库宕机，切换到从库，部分数据丢失。

**分析过程：**
```sql
-- 1. 分析复制模式
SHOW VARIABLES LIKE 'sync_binlog%';

-- sync_binlog = 0
-- 事务提交后不刷盘，系统宕机会丢数据

-- 2. 查看半同步状态
SHOW VARIABLES LIKE 'rpl_semi_sync%';
-- 如果没开启，主库不等待从库确认
```

**根因分析：**
主库未使用半同步复制，宕机时Binlog未同步到从库。

**解决方案：**
1. 开启半同步复制
   ```sql
   -- 主库
   INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
   SET GLOBAL rpl_semi_sync_master_enabled = ON;
   
   -- 从库
   INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
   SET GLOBAL rpl_semi_sync_slave_enabled = ON;
   ```

2. 设置合理的sync_binlog
   ```ini
   sync_binlog = 1  # 每次提交都同步到磁盘
   ```

### 案例3：复制链路断开

**问题现象：**
从库报错，复制中断。

**分析过程：**
```sql
SHOW SLAVE STATUS\G
-- Last_Error: Could not execute Update_rows event
-- errno: 1062  ← 主键冲突

-- 常见错误：
-- 1062: Duplicate entry（主键/唯一键冲突）
-- 1032: Delete row wasn't found（数据不一致）
-- 1213: Deadlock（死锁）
```

**解决方案：**
```sql
-- 1. 跳过错误（临时）
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;

-- 2. GTID模式跳过特定事务
SET GTID_SKIP_TRANSACTION = 'uuid:事务ID';

-- 3. 修复数据不一致
-- pt-table-checksum检查
-- pt-table-sync修复

-- 4. 预防措施
-- 从库禁止写入
super_read_only = ON
-- 使用pt-table-checksum定期检查
```

## 四、高可用方案

### 1. MHA

```bash
# MHA安装
yum install -y mha4mysql-node
yum install -y mha4mysql-manager

# 配置文件
cat > /etc/mha.cnf << 'EOF'
[server default]
manager_workdir=/var/log/mha/app1
manager_log=/var/log/mha/app1/manager.log
remote_workdir=/var/log/mha/app1
ssh_user=root
repl_user=repl
repl_password=xxx
master_binlog_dir=/var/lib/mysql
secondary_binlog_dir=/var/lib/mysql2

[server1]
hostname=master_ip
candidate_master=1

[server2]
hostname=slave1_ip
candidate_master=1

[server3]
hostname=slave2_ip
candidate_master=0
EOF

# 启动MHA
nohup masterha_manager --conf=/etc/mha.cnf &

# 检查集群状态
masterha_check_ssh --conf=/etc/mha.cnf
masterha_check_repl --conf=/etc/mha.cnf
```

### 2. Orchestrator

```bash
# 安装
yum install -y orchestrator

# 配置
cat > /etc/orchestrator.conf.json << 'EOF'
{
  "MySQLTopologyUser": "orchestrator",
  "MySQLTopologyPassword": "xxx",
  "MySQLConnectTimeoutSeconds": 2,
  "DiscoverByIOSession": true,
  "ReadBinlogEventsTimeoutSeconds": 10,
  "ReplicationLagQueryTimeout": 10
}
EOF

# 启动
systemctl start orchestrator

# 使用
orchestrator -c discover master_ip
orchestrator -c topology master_ip
orchestrator -c graceful-master-takeover master_ip
```
