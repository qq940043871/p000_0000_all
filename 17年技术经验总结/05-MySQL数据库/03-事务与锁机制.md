> 更新时间：2026-03-28

## 一、理论基础

### 1. 事务隔离级别

```
隔离级别对比：

┌──────────┬────────────┬────────────┬────────────┬─────────────┐
│ 隔离级别  │ 脏读       │ 不可重复读  │ 幻读        │ 并发性能     │
├──────────┼────────────┼────────────┼────────────┼─────────────┤
│ READ     │ 可能发生   │ 可能发生   │ 可能发生   │ 最高        │
│ UNCOMMITED│            │            │            │             │
├──────────┼────────────┼────────────┼────────────┼─────────────┤
│ READ     │ 不可能    │ 可能发生   │ 可能发生   │ 一般        │
│ COMMITTED │            │            │            │             │
├──────────┼────────────┼────────────┼────────────┼─────────────┤
│ REPEATABLE│ 不可能    │ 不可能     │ 可能发生   │ 较差        │
│ READ     │            │            │            │             │
├──────────┼────────────┼────────────┼────────────┼─────────────┤
│ SERIALIZE│ 不可能    │ 不可能     │ 不可能     │ 最低        │
└──────────┴────────────┴────────────┴────────────┴─────────────┘

脏读：A事务读取B事务未提交的数据，B回滚后数据无效

不可重复读：A事务两次读取同一数据，B事务修改并提交，读取结果不同

幻读：A事务两次查询，B事务新增/删除记录，结果行数不同

MySQL默认：REPEATABLE READ
Oracle默认：READ COMMITTED
```

### 2. MVCC原理

```
MVCC（Multi-Version Concurrency Control）：

每行数据隐藏字段：
  - DB_TRX_ID：最后修改的事务ID
  - DB_ROLL_PTR：指向Undo Log的指针
  - DB_ROW_ID：隐藏主键（无主键时生成）

版本链：
  新事务 → Undo Log ← Undo Log ← Undo Log
           最新数据    历史1      历史2

ReadView（快照读）：
  - m_ids：活动事务ID列表
  - min_trx_id：最小活动事务ID
  - max_trx_id：创建时最大事务ID+1
  - creator_trx_id：当前事务ID

读取规则：
  - trx_id < min_trx_id：已提交，可读
  - trx_id在m_ids中：自己的事务或未提交，看版本链
  - trx_id > max_trx_id：快照之后的事务，不可见

当前读 vs 快照读：
  当前读（SELECT ... FOR UPDATE）：读取最新数据，加锁
  快照读（普通SELECT）：读取历史版本
```

### 3. 锁类型

```
InnoDB锁类型：

行锁（Record Lock）：
  - 锁住索引记录
  - SELECT * FROM users WHERE id = 1 FOR UPDATE;

间隙锁（Gap Lock）：
  - 锁住索引之间的间隙
  - 防止幻读
  - RR级别生效
  SELECT * FROM users WHERE id BETWEEN 1 AND 10 FOR UPDATE;
  → 锁住(1, 10)之间的间隙

临键锁（Next-Key Lock）：
  - 记录锁 + 间隙锁
  - InnoDB默认使用
  - 锁住索引记录及其前面的间隙

意向锁（Intention Lock）：
  - 表级锁
  - IS：意向共享锁（准备获取行锁）
  - IX：意向排他锁（准备获取行锁）
  - 作用：快速判断表是否有行锁

自增锁（AUTO-INC Lock）：
  - 插入自增列时获取
  - 轻量级，快速释放
```

## 二、实践应用

### 1. 锁分析

```sql
-- 查看当前锁
SHOW ENGINE INNODB STATUS\G
-- 关键部分：
-- LOCK WAIT: 锁等待
-- LOCK REC: 行锁信息
-- lock_mode: 锁模式

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
SELECT * FROM information_schema.INNODB_LOCKS;

-- 查看事务
SELECT * FROM information_schema.INNODB_TRX\G

-- 杀掉阻塞的事务
KILL <trx_mysql_thread_id>;
```

### 2. 死锁处理

```sql
-- 查看死锁日志
SHOW ENGINE INNODB STATUS\G
-- TRANSACTION部分：
-- (1) HOLDS THE LOCKS：持有锁
-- (2) WAITING FOR：等待锁
-- 找到互相等待的SQL

-- 死锁自动处理：
-- InnoDB自动回滚undo量小的事务
-- 保留undo量大的事务

-- 避免死锁：
-- 1. 多个事务按相同顺序获取锁
-- 2. 减少事务大小
-- 3. 降低隔离级别
-- 4. 添加合理索引
```

### 3. 隔离级别配置

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION transaction_isolation = 'REPEATABLE-READ';
SET GLOBAL transaction_isolation = 'READ-COMMITTED';

-- InnoDB建议使用RR（默认）
-- 重要场景可用RC（如报表统计）
```

## 三、生产环境问题案例

### 案例1：行锁等待超时

**问题现象：**
更新语句长时间不返回，最后报错"Lock wait timeout exceeded"。

**分析过程：**
```sql
-- 1. 查看当前锁
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
-- 找到waiting_trx_id和blocking_trx_id

-- 2. 查看阻塞事务
SELECT * FROM information_schema.INNODB_TRX 
WHERE trx_id IN (找到的blocking_trx_id);

-- 3. 分析
-- 事务A锁住了行X
-- 事务B等待行X

-- 4. 常见原因：
-- - 长事务未提交
-- - 大事务持有锁
-- - 索引缺失导致锁范围扩大
```

**解决方案：**
```sql
-- 1. 杀掉阻塞事务
KILL <blocking_trx_id>;

-- 2. 优化事务
-- - 拆分大事务
-- - 减少锁定范围
-- - 添加索引减少锁行数

-- 3. 设置超时
SET innodb_lock_wait_timeout = 5;  -- 默认50秒

-- 4. 调整隔离级别
SET SESSION transaction_isolation = 'READ-COMMITTED';
```

### 案例2：间隙锁导致插入失败

**问题现象：**
插入数据时报错，无法插入。

**分析过程：**
```sql
-- 1. 查看错误
-- ERROR 1205 (HY000): Lock wait timeout exceeded

-- 2. 分析
SELECT * FROM users WHERE age BETWEEN 25 AND 30 FOR UPDATE;
-- RR隔离级别，锁住25-30之间所有间隙

-- 3. 其他事务尝试插入
INSERT INTO users (name, age) VALUES ('张三', 27);
-- 被间隙锁阻塞！
```

**根因分析：**
RR级别下，范围查询会锁住所有间隙，防止幻读。

**解决方案：**
1. 缩小锁定范围
2. 降低隔离级别到RC
3. 避免不必要的FOR UPDATE
4. 使用记录锁替代范围锁

```sql
-- 精确锁定
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 只锁1条
-- 比
SELECT * FROM users WHERE age = 25 FOR UPDATE; -- 锁age=25的所有行
```

### 案例3：死锁

**问题现象：**
两个事务互相等待对方持有的锁。

**分析过程：**
```sql
SHOW ENGINE INNODB STATUS\G
-- 显示：
-- TRANSACTION 1:
--   ---LOCK WAIT
--   lock struct(s), heap size 1136, lock...
--   lock_mode X waiting
--   Lock wait time: 10
--   ---TRANSACTION 2:
--   ---HOLDS THE LOCKS
--   lock struct(s), heap size 1136, lock...
--   lock_mode X locks rec but not gap

-- 分析：
-- T1持有A锁，等待B锁
-- T2持有B锁，等待A锁
-- 形成死锁！
```

**解决方案：**
```sql
-- 1. InnoDB自动回滚（undo量小的事务）
-- 2. 业务层重试

-- 3. 预防死锁：
-- - 统一锁获取顺序
BEGIN;
SELECT * FROM users WHERE id=1 FOR UPDATE;  -- 先锁1
SELECT * FROM users WHERE id=2 FOR UPDATE;  -- 再锁2
COMMIT;

-- - 拆分大事务
-- - 减少锁定时间
```

**经验教训：**
- 死锁难以完全避免
- 监控死锁日志
- 业务层做好重试机制
