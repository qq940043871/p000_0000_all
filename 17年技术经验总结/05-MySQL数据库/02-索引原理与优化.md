# 索引原理与优化

> 模块：MySQL数据库
> 更新时间：2026-03-28

---

## 一、理论基础

### 1. B+树结构

```
B+树特点：
  - 所有数据都在叶子节点
  - 叶子节点用双向链表连接（范围查询快）
  - 非叶子节点只存储索引键
  - 树高通常2-4层

B+树 vs B树：
  B树：所有节点都存数据
  B+树：只有叶子节点存数据，非叶子节点只存索引
  → B+树每个节点能存更多索引，树高更低

磁盘IO：
  一次IO读取一个节点（约16KB）
  树高3 = 最多3次IO查到数据
  索引让查询从全表扫描百万次IO → 3-4次IO
```

### 2. InnoDB索引

```
聚簇索引（Clustered Index）：
  - 主键自动创建
  - 数据行的物理存储顺序 = 索引顺序
  - 每个表只有一个聚簇索引
  - 如果没有主键，选择唯一索引
  - 都没有，InnoDB生成隐藏row_id

辅助索引（Secondary Index）：
  - 非主键索引
  - 叶子节点存主键值，不是数据地址
  - 查到后需要回表（查询聚簇索引获取数据）

回表查询：
  SELECT * FROM users WHERE name = '张三'
  
  1. 在name索引树找到'张三'
  2. 获取主键id=123
  3. 用id=123回表查聚簇索引
  4. 返回完整数据行

覆盖索引：
  SELECT id, name FROM users WHERE name = '张三'
  
  1. 在name索引树找到'张三'
  2. 直接返回id和name
  3. 无需回表！（索引包含所有需要的列）
```

### 3. 索引类型

```
B+树索引：
  - 主键索引/唯一索引/普通索引
  - 前缀索引（只索引字符串前N个字符）
  - 复合索引（多列索引）

全文索引（FULLTEXT）：
  - 用于MATCH AGAINST搜索
  - 中文需要分词器（ngram）
  - 5.7+支持InnoDB

空间索引（SPATIAL）：
  - 用于GIS（地理信息系统）
  - 支持POINT、LINESTRING、POLYGON
  - 仅MyISAM支持

哈希索引（Memory引擎）：
  - O(1)查找
  - 不支持范围查询
  - InnoDB自适应哈希索引
```

### 4. 索引匹配规则

```
最左前缀原则（复合索引）：
  索引 INDEX idx(a, b, c)
  
  可以命中：
    WHERE a = 1          ✓ 使用a
    WHERE a = 1 AND b = 2 ✓ 使用a,b
    WHERE a = 1 AND b = 2 AND c = 3 ✓ 使用a,b,c
  
  部分命中：
    WHERE b = 2          ✗ 不使用
    WHERE c = 3          ✗ 不使用
    WHERE b = 2 AND c = 3 ✗ 不使用（无a）
  
  范围查询后断链：
    WHERE a = 1 AND b > 2 AND c = 3
    → a能用，b范围后断，c不能使用

列类型与索引：
  - 小类型优于大类型（int优于varchar）
  - 定长优于变长（char优于varchar）
  - 避免NULL（NULL值难以优化）
```

## 二、实践应用

### 1. 索引分析

```sql
-- 查看查询执行计划
EXPLAIN SELECT * FROM users WHERE name = '张三'\G

-- 关键字段解读：
-- type: ALL=全表扫描, ref=索引查找, range=范围扫描
-- key: 实际使用的索引
-- rows: 预计扫描行数
-- Extra: Using index=覆盖索引, Using filesort=需要排序

-- 强制使用索引
SELECT * FROM users FORCE INDEX(idx_name) WHERE ...

-- 分析索引使用
SHOW INDEX FROM users;

-- 统计索引基数（cardinality）
SHOW INDEX FROM users;
-- 基数低说明区分度差，索引效果不好
```

### 2. 索引优化

```sql
-- 1. 前缀索引（长字符串优化）
ALTER TABLE users ADD KEY (email(10));
-- 只索引前10个字符

-- 2. 复合索引顺序
-- 选择性高的列放前面
-- 例：INDEX idx(status_time(status, create_time))
-- 适合 WHERE status = 1 AND create_time > '2024-01-01'

-- 3. 覆盖索引
-- 创建覆盖索引避免回表
ALTER TABLE orders ADD INDEX idx_user_product (user_id, product_id, status);
-- 现在可以：
SELECT user_id, product_id, status FROM orders 
WHERE user_id = 1;
-- 覆盖索引，无回表

-- 4. 删除冗余索引
-- 索引(a)和索引(a,b) → 索引(a)是冗余的
-- 索引(a,b)和索引(a) → 索引(a)是冗余的
```

### 3. 常见优化场景

```sql
-- 场景1：分页优化
-- 深度分页很慢：LIMIT 10000, 10
-- 优化：记录上一页最后id
SELECT * FROM orders 
WHERE id > #{last_id}
ORDER BY id 
LIMIT 10;

-- 场景2：COUNT优化
-- COUNT(*)很慢，统计大量数据
-- 优化：使用近似值或缓存

-- 场景3：JOIN优化
-- 小表驱动大表
-- 字段类型要一致（避免类型转换）
SELECT * FROM large_table l
INNER JOIN small_table s ON l.type = s.type
-- 确保小表放右边（MySQL JOIN顺序）

-- 场景4：OR条件优化
-- OR不走索引：WHERE a=1 OR b=2
-- 优化1：改为UNION
SELECT * FROM users WHERE a=1
UNION
SELECT * FROM users WHERE b=2;

-- 优化2：使用IN
WHERE a IN (1, 2, 3)
```

## 三、生产环境问题案例

### 案例1：索引失效

**问题现象：**
加了索引，查询还是很慢。

**分析过程：**
```sql
-- 1. 查看执行计划
EXPLAIN SELECT * FROM orders 
WHERE order_no LIKE '%123%'\G

-- type: ALL（全文扫描！）
-- 原因：LIKE以%开头，索引失效

-- 2. 检查函数操作
EXPLAIN SELECT * FROM users 
WHERE DATE(create_time) = '2024-01-01'\G

-- type: ALL
-- 原因：DATE()函数导致索引失效

-- 3. 检查类型转换
EXPLAIN SELECT * FROM users 
WHERE phone = 13800138000\G

-- phone是varchar类型
-- 隐式类型转换，索引失效
```

**解决方案：**
```sql
-- LIKE优化
-- 1. 精确前缀
WHERE order_no LIKE 'PFX123%'  -- 可以用索引

-- 2. 全文索引
ALTER TABLE orders ADD FULLTEXT INDEX ft_order_no(order_no);
SELECT * FROM orders WHERE MATCH(order_no) AGAINST('123');

-- 3. ElasticSearch（海量数据）

-- 函数优化
-- 改用范围查询
WHERE create_time >= '2024-01-01 00:00:00'
AND create_time < '2024-01-02 00:00:00'

-- 类型一致
WHERE phone = '13800138000'  -- 字符串
```

### 案例2：复合索引顺序错误

**问题现象：**
同样查询，有时快有时慢。

**分析过程：**
```sql
-- 索引：INDEX idx(a, b, c)

-- 查询1：快
EXPLAIN SELECT * FROM test WHERE a=1 AND b=2\G
-- key: idx(a, b, c)  ✓

-- 查询2：慢
EXPLAIN SELECT * FROM test WHERE b=2 AND a=1\G
-- 优化器重排后同样用索引，没问题

-- 查询3：慢
EXPLAIN SELECT * FROM test WHERE b=2\G
-- key: NULL  ✗ 不使用索引（缺少a）
```

**根因分析：**
复合索引不满足最左前缀原则。

**解决方案：**
```sql
-- 按最左前缀原则创建索引
-- 根据实际查询调整顺序

-- 场景：查询主要是 a=? 或 a=? AND b=?
-- 索引应该是 (a, b, c)

-- 场景：查询主要是 b=? 或 b=? AND c=?
-- 需要单独建 (b, c) 索引
```

### 案例3：索引区分度低

**问题现象：**
状态字段加了索引，但查询仍然很慢。

**分析过程：**
```sql
-- 查看索引基数
SHOW INDEX FROM orders;
-- Key_name: idx_status
-- Cardinality: 2  （只有2个不同值！）

-- 分析数据分布
SELECT status, COUNT(*) FROM orders GROUP BY status;
-- status=0: 100条
-- status=1: 9999900条

-- 区分度 = 2 / 总行数 ≈ 0
-- 索引完全无效！
```

**根因分析：**
状态字段只有0和1，大量数据集中在某一个值，索引无意义。

**解决方案：**
```sql
-- 方案1：不要在低区分度字段建索引
-- 删除无意义索引

-- 方案2：复合索引提高区分度
-- INDEX idx(status_userid, id)
-- 配合 WHERE status=1 AND user_id=?

-- 方案3：分区表
-- 按状态分区，减少扫描范围

-- 方案4：分表
-- 拆分成 orders_0 和 orders_1
```

**经验教训：**
- 建索引前分析字段区分度
- 区分度 > 20% 才建议建索引
- 监控慢查询，持续优化
