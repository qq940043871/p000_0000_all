## 一、理论基础

### 1. Redis为什么快

```
高性能原因：
  1. 内存存储：所有数据在内存
  2. 单线程：避免上下文切换和锁竞争
  3. IO多路复用：单线程处理高并发
  4. C语言实现：接近硬件

Redis vs Memcached：
  Redis: 支持多种数据结构、持久化、主从复制
  Memcached: 仅支持字符串、分布式但不支持主从
```

### 2. 五种基础数据结构

```
String（字符串）：
  - 底层：SDS（Simple Dynamic String）
  - 用途：缓存、计数器、分布式锁
  - 命令：GET/SET/MGET/MSET/INCR/DECR

Hash（哈希）：
  - 底层：ziplist（小数据）/ hashtable
  - 用途：对象存储、购物车
  - 命令：HSET/HGET/HGETALL/HLEN

List（列表）：
  - 底层：ziplist（小数据）/ linkedlist
  - 用途：消息队列、粉丝列表
  - 命令：LPUSH/RPOP/LRANGE/LEN

Set（集合）：
  - 底层：intset（纯整数）/ hashtable
  - 用途：标签、好友、抽奖
  - 命令：SADD/SMEMBERS/SISMEMBER/SCARD

Sorted Set（有序集合）：
  - 底层：ziplist（小数据）/ skiplist + hashtable
  - 用途：排行榜、延时队列、权重队列
  - 命令：ZADD/ZRANGE/ZSCORE/ZRANK
```

### 3. 底层数据结构

```
SDS（简单动态字符串）：
  - 预分配空间：减少内存分配
  - 惰性释放：字符串缩短时不立即回收
  - 二进制安全：可存储任意二进制

跳跃表（Skip List）：
  - 多层索引实现O(logN)查询
  - Redis独有：红黑树+跳跃表
  - ZSET使用跳跃表

压缩列表（ziplist）：
  - 连续内存，紧凑存储
  - 元素少时自动转换
  - 节省内存但插入/删除开销大

快速列表（quicklist）：
  - linkedlist + ziplist组合
  - 3.2+版本List的实现
  - 平衡内存和性能
```

## 二、实践应用

### 1. 常用命令

```bash
# String操作
SET key value [EX seconds] [PX milliseconds] [NX|XX]
GET key
MGET key1 key2 key3
MSET key1 val1 key2 val2
INCR key          # 原子+1
INCRBY key 100    # 原子+N
DECRBY key 50     # 原子-N
SETNX key value   # 不存在才设置（分布式锁）

# Hash操作
HSET user:1 name "张三" age 25
HGET user:1 name
HGETALL user:1
HINCRBY user:1 age 1

# List操作
LPUSH queue "task1"
RPUSH queue "task2"
LPOP queue
RPOP queue
LRANGE queue 0 -1    # 遍历全部
BRPOP queue 0        # 阻塞等待

# Set操作
SADD tags "java" "redis"
SMEMBERS tags
SISMEMBER tags "java"
SRANDMEMBER tags 3   # 随机取N个

# ZSet操作
ZADD leaderboard 100 "player1"
ZADD leaderboard 200 "player2"
ZREVRANGE leaderboard 0 9  # 前10名
ZSCORE leaderboard "player1"  # 获取分数
ZINCRBY leaderboard 50 "player1"  # 增加分数
```

### 2. 过期策略

```bash
# 设置过期
SET key value EX 3600    # 1小时后过期
EXPIRE key 3600
EXPIREAT key timestamp   # 到时间戳过期
TTL key                  # 查看剩余时间
PTTL key                 # 毫秒级

# 惰性删除 vs 定期删除
惰性删除：访问时检查，过期就删除
定期删除：定时扫描，过期就删除

# 内存淘汰策略
# maxmemory-policy配置
volatile-lru:   仅过期键，使用LRU淘汰
allkeys-lru:    所有键，使用LRU淘汰
volatile-ttl:   仅过期键，淘汰TTL最短的
volatile-random: 仅过期键，随机淘汰
allkeys-random:  所有键，随机淘汰
noeviction:     不淘汰，返回错误
```

### 3. 发布订阅

```bash
# 发布
PUBLISH channel "message"

# 订阅
SUBSCRIBE channel
PSUBSCRIBE pattern*   # 模式订阅

# 取消订阅
UNSUBSCRIBE channel
```

## 三、生产环境问题案例

### 案例1：内存持续增长

**问题现象：**
Redis内存不断增长，没有回落。

**分析过程：**
```bash
# 1. 查看内存使用
redis-cli INFO memory
# used_memory_human: 10G
# used_memory_peak_human: 15G

# 2. 查看大Key
redis-cli --bigkeys

# 3. 分析内存分布
redis-cli MEMORY DOCTOR
```

**根因分析：**
数据只进不出，过期策略未生效。

**解决方案：**
```bash
# 1. 设置过期
SET key value EX 86400

# 2. 配置淘汰策略
# redis.conf
maxmemory 8gb
maxmemory-policy allkeys-lru

# 3. 定期清理
redis-cli --scan | head -1000 | xargs redis-cli UNLINK

# 4. 监控大Key
redis-cli --bigkeys
```

### 案例2：SETNX死锁

**问题现象：**
分布式锁获取失败，业务卡住。

**分析过程：**
```bash
# 1. 查看锁状态
redis-cli GET lock_key
# 发现：锁已存在

# 2. 分析原因
# - 持有锁的服务异常退出，锁未释放
# - 锁超时时间设置不合理
```

**解决方案：**
```python
# 正确的分布式锁
import redis

def acquire_lock(conn, lock_name, timeout=30):
    identifier = str(uuid.uuid4())
    end = time.time() + timeout
    while time.time() < end:
        if conn.set(lock_name, identifier, nx=True, ex=timeout):
            return identifier
        time.sleep(0.001)
    return False

def release_lock(conn, lock_name, identifier):
    # Lua脚本保证原子性
    script = """
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    """
    conn.eval(script, 1, lock_name, identifier)
```

**经验教训：**
- SETNX值要唯一，释放要原子
- 超时时间要合理
- 使用RedLock更安全
