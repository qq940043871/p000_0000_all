## 一、缓存问题

### 案例1：缓存穿透

**问题现象：**
大量请求查询不存在的key，击穿到数据库。

**分析过程：**
```bash
# 查看Redis命中率
redis-cli INFO stats | grep keyspace
# miss很多

# 查看QPS
redis-cli INFO stats | grep instantaneous_ops_per_sec
```

**解决方案：**
```python
# 1. 布隆过滤器
from bloom_filter import BloomFilter
bf = BloomFilter(max_elements=1000000, error_rate=0.01)

def get_data(key):
    if key not in bf:
        return None  # 一定不存在
    return db.get(key)

# 2. 缓存空值
def get_data(key):
    val = redis.get(key)
    if val is None:
        val = db.get(key)
        if val:
            redis.setex(key, 300, val)
        else:
            redis.setex(key, 60, "NULL")  # 缓存空值
    return val if val != "NULL" else None
```

### 案例2：缓存击穿

**问题现象：**
热点key过期，大量请求同时打到数据库。

**分析过程：**
```bash
# 同一时间大量QPS
redis-cli INFO stats | grep instantaneous_ops_per_sec
# 异常高峰
```

**解决方案：**
```python
# 1. 互斥锁
import redis
import time

def get_with_lock(key):
    val = redis.get(key)
    if val:
        return val
    
    lock = redis.set('lock:' + key, 1, nx=True, ex=10)
    if lock:
        val = db.get(key)
        redis.setex(key, 3600, val)
        redis.delete('lock:' + key)
        return val
    else:
        time.sleep(0.1)
        return get_with_lock(key)  # 等待重试

# 2. 永不过期
# value加上版本号/时间戳
# 后台异步更新
```

## 二、性能问题

### 案例4：大Key

**问题现象：**
Redis操作偶尔很慢。

**分析过程：**
```bash
# 查找大Key
redis-cli --bigkeys

# 分析某个Key
redis-cli DEBUG OBJECT ENCODING big_key
redis-cli DEBUG OBJECT IDLETIME big_key

# 扫描大Key
redis-cli --scan --pattern "*" | while read key; do
  size=$(redis-cli --no-raw-memory MEMORY USAGE "$key" 2>/dev/null)
  if [ "$size" -gt 104857600 ]; then  # >100MB
    echo "$key: $size"
  fi
done
```

**解决方案：**
```bash
# 1. 拆分大Key
# 原来：SET user:1:friends "id1,id2,id3,..."
# 改为：SADD user:1:friends id1 id2 id3...

# 2. 压缩
redis-cli --rdb dump.rdb | gzip > dump.rdb.gz

# 3. 定期清理
redis-cli SCAN 0 | while read key; do
  redis-cli UNLINK "$key"
done
```

### 案例5：热点Key

**问题现象：**
某一特定key的QPS极高，单节点扛不住。

**分析过程：**
```bash
# 监控热点Key
redis-cli MONITOR | grep -c "GET hot_key"
```

**解决方案：**
1. 热点Key拆分
```python
# 原来：GET hot_item:12345
# 改为：
shard = hash(hot_item_id) % N
GET hot_item:{item_id}:shard_{shard}
```

2. 本地缓存
```python
# 使用本地缓存热点
local_cache = {}
def get_hot(key):
    if key in local_cache:
        return local_cache[key]
    val = redis.get(key)
    if val:
        local_cache[key] = val
    return val
```

3. 热点探测
```bash
# 使用Redis FPCA算法
redis-cli --latency-history
```

## 三、运维问题

### 案例6：内存碎片

**问题现象：**
used_memory远小于used_memory_rss。

**分析过程：**
```bash
redis-cli INFO memory | grep -E "used_memory|used_memory_rss|mem_fragmentation_ratio"
# mem_fragmentation_ratio > 1.5 说明碎片严重
```

**解决方案：**
```bash
# 1. 重启Redis（最有效）
redis-cli SHUTDOWN NOSAVE && redis-server redis.conf

# 2. 活动碎片整理（4.0+）
redis-cli MEMORY PURGE

# 3. 配置自动整理
activedefrag yes
active-defrag-ignore-bytes 100mb
active-defrag-threshold-lower 10
active-defrag-threshold-upper 100
```
