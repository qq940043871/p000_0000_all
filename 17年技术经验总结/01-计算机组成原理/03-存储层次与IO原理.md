> 模块：计算机组成原理
> 更新时间：2026-03-28

## 一、理论基础

### 1. 存储层次对比

```
存储类型        容量          延迟          价格/TB      读写次数
---------------------------------------------------------------------------
CPU寄存器      <1KB          0.3ns         极高           无限
L1/L2/L3缓存   64KB-64MB     1-40ns        极高           无限
DRAM           GB级          100ns         ~$20           无限
NVMe SSD       TB级          10-100μs      ~$100         有限(写磨损)
SATA SSD       TB级          100-500μs     ~$80          有限
HDD            TB级          5-10ms        ~$20           无限(机械)

关键理解：
- SSD延迟是DRAM的100-1000倍
- HDD延迟是SSD的50-100倍
- 顺序读写 vs 随机读写差距巨大
```

### 2. 磁盘IO原理

#### 机械硬盘（HDD）

```
结构：
  - 盘片（Platter）：铝合金/玻璃材质，表面涂磁性材料
  - 主轴（Spindle）：带动盘片旋转，5400/7200/10000/15000 RPM
  - 磁头（Head）：悬浮在盘片表面，读取/写入数据
  - 磁臂（Actuator Arm）：带动磁头移动
  - 接口：SATA/SAS

寻道时间（Seek Time）：
  - 磁头移动到正确磁道的平均时间
  - 7200RPM HDD：约9ms
  - 10000RPM HDD：约5ms
  - 15000RPM HDD：约3.5ms

旋转延迟（Rotational Latency）：
  - 等待目标扇区转到磁头下方的时间
  - 7200RPM：平均4.2ms（1/2转速）
  - 10000RPM：平均3ms

IO时间计算：
  IO时间 = 寻道时间 + 旋转延迟 + 传输时间
         = 9ms      + 4ms        + 传输时间（可忽略）
         ≈ 13ms（对于随机IO）
```

#### 固态硬盘（SSD）

```
结构：
  - NAND Flash：存储介质
  - 主控芯片：FTL（Flash Translation Layer）转换
  - DRAM缓存：缓存映射表和热点数据
  - 接口：SATA/SAS/NVMe

Flash特性：
  - Page（页）：最小读写单元，4KB/8KB
  - Block（块）：最小擦除单元，256KB-4MB
  - 写入前必须擦除，擦除以Block为单位
  - 写放大（Write Amplification）：有效写入量 > 实际数据量

寿命：
  - SLC：100000次擦写
  - MLC：3000-10000次
  - TLC：1000-3000次
  - QLC：300-1000次

ECC纠错：
  - 每页有冗余数据用于纠错
  - 错误率随使用时间增加
```

### 3. RAID技术

```
RAID级别对比：

RAID 0（条带化）
  - 原理：数据分散写入多个磁盘
  - 优点：读写性能提升n倍
  - 缺点：无冗余，1块盘坏数据全丢
  - 可用容量：N x 单盘容量
  - 适用：临时数据、高性能要求

RAID 1（镜像）
  - 原理：数据同时写入两个磁盘
  - 优点：冗余，坏1块盘不影响
  - 缺点：容量利用率50%
  - 可用容量：1 x 单盘容量
  - 适用：系统盘、重要数据

RAID 5（奇偶校验）
  - 原理：数据+校验码分散存储
  - 优点：允许坏1块盘，读性能好
  - 缺点：写性能差，重建慢
  - 可用容量：(N-1) x 单盘容量
  - 适用：通用存储、数据库

RAID 6（双重校验）
  - 原理：两组独立校验码
  - 优点：允许坏2块盘，安全性高
  - 缺点：容量开销更大
  - 可用容量：(N-2) x 单盘容量
  - 适用：大规模存储、归档

RAID 10（1+0，先镜像后条带）
  - 原理：RAID1镜像后再条带化
  - 优点：读写性能好，允许坏多块（不同组）
  - 缺点：容量利用率50%
  - 可用容量：N/2 x 单盘容量
  - 适用：数据库、高IOPS需求

RAID 60（6+0）
  - 原理：RAID6组再条带化
  - 优点：高可靠+高性能
  - 缺点：需要至少8块盘
  - 适用：大型视频/备份系统
```

### 4. IO调度算法

```
Linux IO调度器演进：
  - CFQ（Completely Fair Queuing）：3.12之前默认，公平但延迟高
  - Deadline：限制延迟，适合数据库
  - NOOP：不做排序，适合SSD
  - mq-deadline/mq-NOOP：多队列版本（4.19+默认）

查看/修改调度器：
  # 查看当前调度器
  cat /sys/block/sda/queue/scheduler

  # 临时修改
  echo "deadline" > /sys/block/sda/queue/scheduler

  # 永久修改（SSD）
  echo "noop" > /etc/udev/rules.d/60-ssd.rules
```

## 二、实践应用

### 1. 磁盘性能测试

```bash
# 顺序读测试
fio --name=read_test --ioengine=libaio --rw=read \
    --bs=1m --size=1g --numjobs=4 --runtime=60 \
    --filename=/dev/sda --group_reporting

# 顺序写测试
fio --name=write_test --ioengine=libaio --rw=write \
    --bs=1m --size=1g --numjobs=4 --runtime=60 \
    --filename=/dev/sda --group_reporting

# 随机读写测试
fio --name=randrw --ioengine=libaio --rw=randrw \
    --bs=4k --size=1g --numjobs=4 --runtime=60 \
    --filename=/dev/sda --group_reporting

# 简单dd测试
dd if=/dev/zero of=test bs=1M count=1024 oflag=direct
dd if=test of=/dev/null bs=1M iflag=direct
```

### 2. IO监控工具

```bash
# 实时IO监控
iostat -xz 1

# 详细统计
iostat -dx 1
# 关键指标：
#   %util：IO使用率，>80%说明IO饱和
#   avgrq-sz：平均请求大小
#   avgqu-sz：平均队列长度
#   await：平均响应时间

# 进程IO统计
iotop
# 或
pidstat -d 1

# 查看IO等待
vmstat 1
# wa列：CPU等待IO时间，>20%说明IO瓶颈
```

### 3. 文件系统选择

```
场景 vs 文件系统：

高并发小文件（图片、静态资源）
  → ext4 + noatime
  → 或 XFS（大文件更好）

数据库（MySQL/PostgreSQL）
  → XFS 或 ext4
  → InnoDB使用原始分区或指定数据目录
  → 注意：关闭atime/noatime，减少写入

大文件存储（视频、备份）
  → XFS（最大16EB，单文件16TB）
  → 或 ZFS（高级特性，但内存需求高）

高可靠性（企业存储）
  → ext4（成熟稳定）
  → 或 btrfs（快照、校验，但生产环境慎用）

性能测试对比：
  → fio测试不同文件系统在实际负载下的表现
```

### 4. Linux IO优化参数

```bash
# 查看当前值
cat /proc/sys/vm/dirty_ratio       # 内存中脏页比例
cat /proc/sys/vm/dirty_background_ratio
cat /proc/sys/vm/dirty_expire_centisecs

# 优化配置（/etc/sysctl.conf）
vm.dirty_ratio = 15
vm.dirty_background_ratio = 5
vm.dirty_expire_centisecs = 500
vm.dirty_writeback_centisecs = 100

# SSD优化
# 1. 启用TRIM
fstrim -av

# 2. IO调度器
echo "noop" > /sys/block/sda/queue/scheduler

# 3. 纳秒时间戳（减少系统调用开销）
echo 0 > /proc/sys/kernel/hz_random
```

## 三、生产环境问题案例

### 案例1：IO瓶颈导致数据库响应慢

**问题现象：**
MySQL查询突然变慢，大量请求超时。

**分析过程：**
```bash
# 1. 查看整体IO状态
iostat -xz 1
# 发现：sdb磁盘%util=100%，avgqu-sz=50+

# 2. 查看哪些进程在IO
iotop -o
# 发现：备份进程在做大量顺序写

# 3. 分析
# 备份进程带宽占用，导致用户请求IO排队
```

**根因分析：**
备份进程和业务IO竞争磁盘带宽，业务IO排队等待。

**解决方案：**
1. 错峰备份：业务低峰期执行
2. 使用 ionice 设置备份进程低优先级
   ```bash
   ionice -c 3 -p $(pgrep backup)
   ```
3. 独立磁盘：备份使用单独的物理磁盘
4. 限速：使用 cgroup 限制备份IO

**经验总结：**
- IO密集型服务要独立磁盘
- 备份任务使用低IO优先级
- 监控磁盘%util和队列长度

### 案例2：SSD写放大导致性能骤降

**问题现象：**
新上线的SSD，运行几个月后写入速度明显下降。

**分析过程：**
```bash
# 查看SSD SMART信息
smartctl -a /dev/nvme0n1

# 查看写放大情况
cat /sys/block/nvme0n1/nvme_write_granularity
cat /sys/block/nvme0n1/nvme_write_cache

# 测试顺序写入速度
fio --name=seq_write --rw=write --bs=1m --size=10g --numjobs=1
# 初始：1GB/s，几个月后：200MB/s
```

**根因分析：**
大量小文件随机写入，导致严重的写放大效应。SSD空闲块减少后，新写入需要先擦除再写入。

**解决方案：**
1. 开启TRIM：让SSD及时回收空间
   ```bash
   fstrim -av  # 定期执行
   ```
2. 优化写入模式：合并小写为大顺序写
3. 预留空间：SSD保留10-20%空闲空间
4. 更换SSD：写入量接近寿命

**经验总结：**
- SSD有写入寿命和写放大问题
- 监控SMART信息的可用余量
- 大量小文件场景考虑内存文件系统

### 案例3：RAID卡缓存导致数据丢失风险

**问题现象：**
服务器意外断电后，部分数据丢失。

**分析过程：**
```bash
# 查看RAID卡配置
MegaCli -LDInfo -Lall -aAll

# 发现：写入策略为 WriteBack（写缓存）
# 问题：BBU（电池）损坏或放电时，写缓存被禁用，性能骤降
# 更严重：保护电容失效时，写缓存中的数据可能丢失

# 检查BBU状态
MegaCli -AdpBbuCmd -aAll
# 发现：BBU需要充电/已损坏
```

**根因分析：**
RAID卡使用WriteBack模式但BBU失效，掉电时缓存数据未写入磁盘。

**解决方案：**
1. 立即检查BBU状态
2. BBU损坏期间切换为WriteThrough模式
3. 更换BBU或启用Cachingvall（三星电容）
4. 对重要数据使用fsync保证

**经验总结：**
- RAID卡缓存需要电池保护
- 定期检查RAID卡硬件状态
- 关键业务不能依赖RAID卡缓存保证数据安全
- 使用UPS
