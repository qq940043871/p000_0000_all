# 内存管理与缓存机制

> 模块：计算机组成原理
> 更新时间：2026-03-28

## 一、理论基础

### 1. 内存层级

```
寄存器 → L1 Cache → L2 Cache → L3 Cache → RAM → SSD/HDD

访问延迟对比：
  寄存器：    ~1 ns
  L1 Cache：  ~4 ns
  L2 Cache：  ~12 ns
  L3 Cache：  ~40 ns
  RAM：       ~100-200 ns
  NVMe SSD：  ~10-100 μs
  SATA SSD：  ~100-500 μs
  HDD：       ~5-10 ms

距离感理解：
  - L1 Cache访问 ≈ 从CPU到桌面拿一张纸的时间
  - 内存访问 ≈ 出门走一个街区的时间
  - SSD访问 ≈ 跨城市快递
  - HDD访问 ≈ 坐飞机去另一个国家
```

### 2. 缓存行与内存对齐

```
缓存行（Cache Line）
  - CPU和内存交换的最小单位
  - 大小：64字节（现代CPU主流）

内存对齐示例：
  // 未对齐：需要两次内存访问
  struct A {
    char a;    // 1字节
    int b;     // 4字节（需要对齐到4字节边界）
    char c;    // 1字节
  };  // sizeof(A) = 12（实际占用）

  // 对齐后：一次内存访问
  struct B {
    int b;     // 4字节（对齐到起始位置）
    char a;    // 1字节
    char c;    // 1字节
    char pad[2]; // 2字节填充
  };  // sizeof(B) = 8
```

### 3. 缓存策略

```
写策略：
  - Write Through（写直达）：同时写缓存和主存
  - Write Back（写回）：只写缓存，淘汰时写回主存

读策略：
  - Write Allocate（写分配）：写入未命中时，先加载到缓存，再更新
  - No-Write Allocate（写不分配）：直接写主存，不加载到缓存

组合：
  - WT + No-WA：直接写主存，速度快但一致性差
  - WB + WA：现代CPU主流选择
```

### 4. TLB（Translation Lookaside Buffer）

```
作用：加速虚拟地址到物理地址的转换

工作原理：
  虚拟地址 → TLB查找 → 命中 → 物理地址
                 ↓ 未命中
              页表查找 → 更新TLB → 物理地址

TLB结构：
  - 全关联/组相联
  - 条目数：几十到几百
  - 命中率：95%+（热点数据）

TLBmiss代价：
  - 需要访问内存中的页表
  - 额外几十到上百个时钟周期
```

### 5. 大页内存（HugePages）

```
标准页大小：4KB
大页大小：2MB / 1GB（需要硬件支持）

使用场景：
  - 数据库（Oracle、PostgreSQL）
  - Java虚拟机（大堆内存）
  - 高性能计算

优势：
  - 减少TLB条目压力
  - 降低页表内存占用
  - 减少换页

配置示例（Linux）：
  # 临时设置
  echo 1024 > /proc/sys/vm/nr_hugepages

  # 永久设置
  vim /etc/sysctl.conf
  vm.nr_hugepages = 1024
```

## 二、实践应用

### 1. 查看内存信息

```bash
# 查看内存总量和使用
free -h

# 详细内存信息
cat /proc/meminfo

# 查看内存硬件信息
dmidecode -t memory

# 查看NUMA节点内存
numactl --hardware

# 查看大页配置
cat /proc/meminfo | grep -i huge
```

### 2. 内存分析工具

```bash
# 实时内存监控
top / htop

# 进程内存映射
pmap -x <pid>

# 详细内存统计
cat /proc/<pid>/status | grep -E "Vm|Rss"

# 内存泄漏检测
valgrind --leak-check=full ./program
# 或
gdb attach <pid>
  (gdb) set pagination off
  (gdb) info registers
  (gdb) bt full
```

### 3. 缓存分析

```bash
# 查看缓存信息
cat /sys/devices/system/cpu/cpu0/cache/index*/size
cat /sys/devices/system/cpu/cpu0/cache/index*/type

# 缓存带宽测试
bw_mem 1G rd
bw_mem 1G wr

# 内存带宽测试
stream
```

### 4. 内存压力测试

```bash
# 简单内存写入测试
dd if=/dev/zero of=/dev/shm/test bs=1M count=1000
# 写文件系统缓存
dd if=/dev/zero of=/tmp/test bs=1M count=1000

# stress工具压力测试
stress --vm 2 --vm-bytes 80% --timeout 60s
```

## 三、生产环境问题案例

### 案例1：内存泄漏排查

**问题现象：**
Java服务运行几天后内存持续增长，最终OOM被杀死。

**分析过程：**
```bash
# 1. 确认是否真的泄漏
# 观察内存曲线：稳定 vs 持续增长

# 2. 生成堆快照
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 使用MAT分析
java -jar mat/HeapAnalyzer.jar heap.hprof

# 4. 在线分析
jmap -histo <pid> | head -50
# 查看对象数量和大小

# 5. GC日志分析
-XX:+PrintGCDetails -Xloggc:gc.log
```

**根因分析：**
静态集合类（HashMap/ArrayList）持续添加对象，无清理逻辑。

**解决方案：**
1. 短期：重启服务 + 增加监控
2. 长期：代码审查，修复泄漏点
3. 添加Old Gen告警，提前发现

**经验总结：**
- 内存泄漏特征：稳定增长，不回落
- 静态集合、监听器、未关闭连接是常见原因
- 定期生成堆快照对比

### 案例2：缓存污染导致性能下降

**问题现象：**
数据处理程序，刚启动时快，运行一段时间后变慢。

**分析过程：**
```bash
# 使用perf分析缓存命中率
perf stat -e cache-misses,cache-references ./program

# 发现：cache-misses率高达40%
# 原因：处理大数据集时，将冷数据冲刷了热数据

# 进一步分析
perf record -e cache-misses -g ./program
perf report
```

**根因分析：**
LRU缓存策略被大文件扫描污染，热数据被淘汰。

**解决方案：**
1. 分批处理数据，避免一次性扫描
2. 使用LFU替代LRU策略
3. 使用内存池，复用热数据

**经验总结：**
- 缓存污染是常见性能陷阱
- 大数据处理要注意局部性原理
- 监控缓存命中率

### 案例3：Swap导致系统卡死

**问题现象：**
服务器内存不足，开始使用Swap，系统几乎完全卡死。

**分析过程：**
```bash
# 查看swap使用
free -h
swapon -s

# 查看谁在使用内存
for i in $(ls /proc/ | grep '^[0-9]*$'); do
  if [ -d /proc/$i ]; then
    rss=$(grep VmRSS /proc/$i/status 2>/dev/null | awk '{print $2}')
    if [ -n "$rss" ] && [ $rss -gt 1024000 ]; then
      echo "PID: $i RSS: $rss KB"
    fi
  fi
done

# 关闭swap测试
swapoff -a
```

**根因分析：**
Java服务设置了过大的堆内存（-Xmx32g），物理内存不足时触发Swap。

**解决方案：**
1. 降低堆内存大小，预留足够OS内存
2. 设置vm.swappiness参数，降低Swap倾向
3. 增加物理内存
4. 容器环境设置内存限制

**经验总结：**
- Java堆不要超过物理内存的70%
- 设置swappiness=10或更低
- 生产环境监控Swap使用率
