## 一、常见问题速查

### 1. CPU相关

| 问题现象 | 可能原因 | 排查命令 | 解决方案 |
|---------|---------|---------|---------|
| CPU 100% | 死循环、正则、加密 | `top -H -p pid` | jstack分析、代码优化 |
| CPU 波动大 | 定时任务、资源竞争 | `perf top` | 错峰、扩容 |
| 多核不均衡 | NUMA、绑定问题 | `numastat` | 绑定CPU亲和性 |
| 软中断高 | 网卡队列瓶颈 | `cat /proc/softirqs` | 调整网卡队列 |

### 2. 内存相关

| 问题现象 | 可能原因 | 排查命令 | 解决方案 |
|---------|---------|---------|---------|
| 内存持续增长 | 泄漏、缓存无限增长 | `pmap -x pid`、`jmap` | 代码修复、重启 |
| OOM被杀死 | 内存分配超限 | `dmesg | grep OOM` | 降低堆/增加内存 |
| Swap开始使用 | 内存不足 | `free -h`、`vmstat` | 增加内存、优化使用 |
| 内存碎片化 | 长期运行 | `cat /proc/buddyinfo` | 重启、内存池 |

### 3. IO相关

| 问题现象 | 可能原因 | 排查命令 | 解决方案 |
|---------|---------|---------|---------|
| IO延迟高 | 磁盘繁忙 | `iostat -xz 1` | 错峰、独立磁盘 |
| IO等待高 | CPU等待IO | `vmstat 1`（wa列） | 优化IO、换SSD |
| 写放大 | SSD写入策略 | `smartctl` | TRIM、合并写入 |
| 队列积压 | IOPS超限 | `iostat -x`（avgqu-sz） | 扩容、限流 |

## 二、综合排查方法论

### 1. 快速定位法

```
第一步：确认是哪个层面
  CPU高？ → 系统层 vs 进程层
  内存高？ → 应用层 vs 系统层
  IO高？ → 读？写？顺序？随机？

第二步：缩小范围
  系统层 → dmesg、top、vmstat
  进程层 → pidstat、jstack、pmap

第三步：深入分析
  CPU → perf、火焰图
  内存 → 堆快照、内存分配
  IO → iostat、blktrace

第四步：验证假设
  修改参数 → 观察效果
```

### 2. 常用排查命令速查

```bash
# 系统整体状态
uptime                    # 负载、运行时间
top                       # CPU、内存、进程
vmstat 1                  # CPU、内存、IO、Swap
iostat -xz 1              # 磁盘IO详情
free -h                   # 内存使用
df -h                     # 磁盘空间

# 进程详情
ps aux | head -20         # 进程列表
pstree -p <pid>           # 进程树
lsof -p <pid>             # 打开的文件
pmap -x <pid>             # 内存映射
strace -p <pid>           # 系统调用
cat /proc/<pid>/status     # 进程状态

# 网络相关
netstat -tunlp            # 监听端口
ss -s                     # 连接统计
netstat -ant | awk '/^tcp/ {s[$NF]++} END {for(k in s) print k, s[k]}'

# 日志
dmesg | tail              # 内核日志
journalctl -xe            # systemd日志
/var/log/messages         # 系统日志
```

## 三、经典案例汇总

### 案例1：Java服务频繁Full GC

**问题描述：**
服务运行2-3天后开始变慢，GC日志显示频繁Full GC。

**排查过程：**
```bash
# 1. 查看GC日志
cat gc.log | grep Full

# 2. 分析GC原因
# - Full GC次数多
# - 老年代增长快
# - 对象直接进入老年代

# 3. 堆内存分析
jmap -histo <pid> | head -30
jmap -dump:format=b,file=heap.hprof <pid>

# 4. MAT分析
# 发现：HashMap持续增长，未清理
```

**解决方案：**
```java
// 问题代码
private static Map<String, Object> cache = new HashMap<>();
// 持续添加，从不清理

// 修复方案1：使用WeakHashMap
private static Map<String, Object> cache = new WeakHashMap<>();

// 修复方案2：定期清理
ScheduledExecutorService cleaner = Executors.newSingleThreadScheduledExecutor();
cleaner.scheduleAtFixedRate(() -> cache.clear(), 1, 1, TimeUnit.HOURS);

// 修复方案3：使用Guava Cache
Cache<String, Object> cache = CacheBuilder.newBuilder()
    .maximumSize(10000)
    .expireAfterAccess(1, TimeUnit.HOURS)
    .build();
```

**经验教训：**
- 静态集合是内存泄漏重灾区
- 使用WeakHashMap或带过期时间的缓存
- 添加Old Gen监控告警

### 案例2：Nginx请求堆积

**问题描述：**
Nginx响应慢，部分请求超时504。

**排查过程：**
```bash
# 1. 查看Nginx状态
curl http://127.0.0.1/nginx_status

# active connections: 291
# server accepts handled requests
# 16630948 16630948 31070465
# Reading: 6 Writing: 179 Waiting: 106

# Writing: 179 表示正在处理
# 如果持续很高，说明后端慢

# 2. 查看连接状态
netstat -an | awk '/^tcp/ {s[$6]++} END {for(k in s) print k, s[k]}'

# 3. 查看后端PHP-FPM状态
curl http://127.0.0.1/php_status

# 4. 查看PHP-FPM进程
pmap -x <php-fpm-pid> | tail -5
```

**根因：**
PHP-FPM进程内存不足，被OOM Killer杀死，新请求无法处理。

**解决方案：**
```nginx
# nginx.conf 限流
limit_req_zone $binary_remote_addr zone=one:10m rate=10r/s;
server {
    limit_req zone=one burst=50;
}
```

```php
# php.ini 优化
pm.max_children = 50        # 最大子进程
pm.start_servers = 10       # 启动进程
pm.min_spare_servers = 5    # 最小空闲
pm.max_requests = 500       # 回收前处理请求数
```

**经验教训：**
- 监控Nginx Writing状态
- PHP-FPM合理配置进程数
- 内存泄漏会导致进程异常

### 案例3：Redis主从切换后数据丢失

**问题描述：**
Redis主节点宕机，自动切换到从节点，部分数据丢失。

**排查过程：**
```bash
# 1. 查看Redis配置
redis-cli info replication

# 2. 查看主从延迟
redis-cli info | grep lag
# lag: 0 表示无延迟

# 3. 查看AOF配置
redis-cli config get appendonly
redis-cli config get appendfsync

# 4. 分析
# appendfsync = everysec（每秒一次，可能丢1秒数据）
# 主节点宕机时，缓冲区数据可能未同步
```

**根因：**
AOF配置为everysec，异步刷盘，宕机可能丢1秒数据。

**解决方案：**
```redis
# 方案1：always模式（性能损失，数据安全）
config set appendfsync always

# 方案2：everysec + 从节点提升（生产常用）
# 接受1秒数据丢失，但保证性能

# 方案3：混合持久化
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

# 方案4：增加从节点延迟切
# 配置从节点等待主节点恢复，而非立即提升
```

**经验教训：**
- Redis不保证零数据丢失（除非always模式）
- 关键数据要有备份
- 主从切换要谨慎评估

## 四、性能优化checklist

### 系统层面
```bash
# 内核参数优化
cat >> /etc/sysctl.conf << 'EOF'
# 网络优化
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535
net.ipv4.tcp_fin_timeout = 30
net.ipv4.tcp_keepalive_time = 1200

# 文件描述符
fs.file-max = 1000000
fs.nr_open = 1000000

# 内存
vm.swappiness = 10
vm.dirty_ratio = 60
vm.dirty_background_ratio = 5
EOF

sysctl -p
```

### 服务层面
```bash
# /etc/security/limits.conf
* soft nofile 1000000
* hard nofile 1000000
* soft nproc 65535
* hard nproc 65535

# /etc/profile
ulimit -n 1000000
```

## 五、监控告警指标

### 必监控项
```
CPU：
  - CPU使用率 > 80% 持续5分钟 → 告警
  - Load Average > CPU核数 → 告警
  - 软中断占比 > 30% → 告警

内存：
  - 内存使用率 > 85% → 告警
  - Swap使用率 > 20% → 告警
  - 特定进程内存增长 > 50%/小时 → 告警

IO：
  - 磁盘使用率 > 85% → 告警
  - IO使用率 > 80% → 告警
  - IO队列长度 > 32 → 告警
```
