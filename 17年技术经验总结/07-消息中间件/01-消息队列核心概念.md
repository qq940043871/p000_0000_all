# 消息队列核心概念

> 模块：消息中间件
> 更新时间：2026-03-28

## 一、理论基础

### 1. 为什么需要消息队列

```
解耦：
  A → B，A不需要知道B的细节
  只需知道消息格式

异步：
  同步调用耗时100ms
  异步投递耗时5ms
  剩下的时间做别的事

削峰：
  请求突增 → 队列缓冲 → 系统稳定处理
  10000请求/秒 → 队列 → 100请求/秒处理

最终一致性：
  分布式事务 → 消息 → 补偿 → 最终一致
```

### 2. 消息模型

```
点对点（Point-to-Point）：
  ┌─────┐    消息     ┌─────┐
  │ P1  │ ─────────► │Queue│ ────► Consumer
  │ P2  │ ─────────► │     │ ────► Consumer
  └─────┘            └─────┘
  多个生产者          每条消息只被一个消费者消费

发布/订阅（Pub/Sub）：
  ┌─────┐    消息     ┌────────┐
  │Pub1 │ ──────────►│ Topic  │ ────► Consumer1
  │Pub2 │ ──────────►│        │ ────► Consumer2
  └─────┘            └────────┘
  多个发布者          每条消息被所有订阅者消费
```

### 3. 投递语义

```
At Most Once（最多一次）：
  - 不保证消息一定到达
  - 可能丢失消息
  - 适合：日志、统计

At Least Once（至少一次）：
  - 消息不会丢失
  - 但可能重复
  - 适合：大部分业务

Exactly Once（恰好一次）：
  - 消息恰好处理一次
  - 实现复杂，需要幂等
  - 适合：支付、订单
```

### 4. 消息顺序

```
全局有序：
  - 所有消息按顺序消费
  - 性能差，很少使用

部分有序（分区/队列）：
  - 同一分区/队列内有序
  - 不同分区无序
  - 常见实现

消费顺序：
  单消费者 → 顺序消费
  多消费者 → 无序（除非用队列隔离）
```

## 三、Kafka核心概念

```
主题（Topic）：
  - 消息的逻辑分类
  - 可分区（Partition）

分区（Partition）：
  - 物理存储单元
  - 副本（Replica）保证高可用
  - Leader负责读写，Follower同步

消费者组（Consumer Group）：
  - 一组消费者共同消费
  - 分区内负载均衡
  - 不同组消费全部消息

Offset：
  - 消费者消费到哪条消息
  - 可手动提交
  - 支持回溯

ISR（In-Sync Replicas）：
  - 与Leader同步的副本集合
  - ISR列表之外的副本不同步
```
