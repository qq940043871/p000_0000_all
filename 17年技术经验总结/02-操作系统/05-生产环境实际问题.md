## 一、综合排查方法论

### 1. 系统性能分析框架

```
top命令解读：
  - 第1行：uptime、登录用户数、load average
  - 第2行：进程状态统计（运行/睡眠/停止/僵尸）
  - 第3行：CPU使用率（us/sy/ni/id/wa/hi/si/st）
  - 第4/5行：内存使用（total/free/used/buffers/cache）

Load Average解读：
  - 单核CPU：load<1 正常，>1 繁忙
  - 4核CPU：load<4 正常，>4 繁忙
  - 持续>核数：需要扩容或优化

CPU各状态含义：
  us：用户空间进程占用
  sy：内核空间占用
  ni：nice调整的进程
  id：空闲
  wa：等待IO（重要！>20%说明IO瓶颈）
  hi：硬件中断
  si：软件中断
  st：虚拟化盗用（宿主机）
```

### 2. 进程状态速查

```bash
# 进程状态
R：运行中
S：可中断睡眠（等待IO/信号）
D：不可中断睡眠（等待IO）
T：暂停/跟踪
Z：僵尸进程
X：死进程（已释放）

# D状态进程分析
# 通常是IO等待或网络等待
ps -eo pid,stat,wchan:30,cmd | grep "^.* D "

# 僵尸进程分析
ps aux | grep Z
# 查找父进程
ps -ef | grep <parent_pid>
# 解决方案：重启父进程或kill -9
```

### 3. 性能分析工具链

```bash
# 快速定位
uptime            # 负载
top               # 整体CPU/内存
vmstat 1          # 详细系统统计
iostat -xz 1      # IO分析
mpstat -P ALL 1   # 每个CPU核心

# 深入分析
perf top          # CPU热点分析
perf record -g    # 采样记录
strace -p <pid>   # 系统调用追踪
ltrace -p <pid>  # 库调用追踪

# 网络分析
ss -s             # 连接统计
netstat -ant      # 连接状态
tcpdump -i eth0   # 抓包

# 内存分析
free -h           # 内存使用
pmap -x <pid>     # 进程内存映射
cat /proc/meminfo # 详细内存信息
```

## 二、经典案例汇总

### 案例1：Load高但CPU空闲

**问题描述：**
服务器Load Average持续很高（20+），但CPU使用率很低（10%）。

**排查过程：**
```bash
# 1. 确认Load含义
uptime
# load average: 22.34, 20.12, 18.56

# 2. 查看CPU
mpstat -P ALL 1
# CPU空闲率很高

# 3. 查看进程状态
ps aux | awk '$8 ~ /D/ {print}'
# 发现大量D状态进程

# 4. 查看IO
iostat -xz 1
# disk sdb: %util=100%, avgqsz=256
# 大量IO等待

# 5. 结论：IO瓶颈导致进程阻塞，Load升高
```

**解决方案：**
1. 找出IO来源：iotop
2. 优化或限制IO密集型任务
3. 升级到SSD
4. 使用ionice降低优先级

---

### 案例2：内存不足触发OOM Killer

**问题描述：**
服务进程被杀死，dmesg显示OOM Killer。

**排查过程：**
```bash
# 1. 查看OOM日志
dmesg | grep -i "out of memory"
dmesg | grep -i "killed process"

# 2. 分析oom_score
cat /proc/<pid>/oom_score
cat /proc/<pid>/oom_score_adj

# 3. 查看内存分配
free -h
cat /proc/meminfo | grep -E "MemAvailable|MemFree|SwapTotal|SwapFree"

# 4. 查找内存大户
ps aux --sort=-%mem | head -10
```

**解决方案：**
```bash
# 1. 降低内存使用
# Java: -Xmx4g -Xms4g

# 2. 限制容器内存
docker run -m 8g ...

# 3. 设置oom_score_adj
echo -1000 > /proc/<pid>/oom_score_adj  # 降低被杀概率

# 4. 增加物理内存
```

### 案例3：文件描述符耗尽

**问题描述：**
应用报错"Too many open files"。

**排查过程：**
```bash
# 1. 查看当前限制
ulimit -n
cat /proc/sys/fs/file-max

# 2. 查看使用量
cat /proc/sys/fs/file-nr
# allocated   free   allocated  max
# 327680    327680   327680    1000000

# 3. 查看进程
lsof -p <pid> | wc -l
# 或
ls /proc/<pid>/fd | wc -l

# 4. 找出打开的文件
lsof -p <pid> | awk '{print $NF}' | sort | uniq -c | sort -rn | head
```

**解决方案：**
```bash
# 1. 临时提高
ulimit -n 65535

# 2. 永久修改
# /etc/security/limits.conf
* soft nofile 65535
* hard nofile 65535

# 3. 应用层面
# 检查是否泄漏：文件/连接/线程
```

### 案例4：僵尸进程过多

**问题描述：**
系统出现大量僵尸进程。

**排查过程：**
```bash
# 1. 查看僵尸进程
ps aux | awk '$8 == "Z" {print}'
# 或
ps -ef | grep -w Z

# 2. 查找父进程
# 僵尸进程的PPID通常是1（init）或某个服务

# 3. 分析原因
# 父进程未调用wait()回收子进程
# 常见原因：
#   - 父进程死循环未退出
#   - 父进程异常未处理SIGCHLD
#   - fork bomb导致资源耗尽
```

**解决方案：**
1. 重启父进程
2. kill -9父进程（会带走所有子进程）
3. 代码修复：正确处理SIGCHLD信号
   ```c
   signal(SIGCHLD, SIG_IGN);  // 忽略，子进程自动回收
   // 或
   signal(SIGCHLD, handler);  // 自定义处理
   ```

### 案例5：定时任务不执行

**问题描述：**
crontab定时任务没有执行。

**排查过程：**
```bash
# 1. 查看cron服务状态
systemctl status crond
service crond status

# 2. 查看crontab
crontab -l
cat /etc/crontab

# 3. 查看cron日志
tail -f /var/log/cron
# 或
journalctl -u crond -f

# 4. 常见原因：
#   - 环境变量问题
#   - 脚本权限问题
#   - PATH问题
#   - 脚本中有交互输入
```

**解决方案：**
```bash
# 1. 使用绝对路径
0 * * * * /usr/bin/python3 /opt/scripts/backup.py

# 2. 设置环境变量
0 * * * * source /etc/profile; /opt/scripts/backup.sh

# 3. 输出日志
0 * * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# 4. 测试脚本
/usr/bin/python3 /opt/scripts/backup.py
```

## 三、调优checklist

### 系统层面
```bash
# /etc/sysctl.conf 必配
vm.swappiness = 10
vm.overcommit_memory = 1
net.core.somaxconn = 65535
fs.file-max = 1000000

# /etc/security/limits.conf 必配
* soft nofile 65535
* hard nofile 65535
```

### 服务层面
```bash
# JVM
-Xmx: 不超过物理内存70%
-XX:+UseG1GC
-XX:MaxRAMPercentage=70.0

# Nginx
worker_processes auto
worker_connections 65535
worker_rlimit_nofile 65535

# MySQL
innodb_buffer_pool_size: 物理内存70%
max_connections: 根据实际调整
```

## 四、监控告警指标

### 必监控
```
CPU：
  - Load > 核心数 → 告警
  - iowait > 20% → 告警
  - CPU使用率 > 80% 持续5分钟 → 告警

内存：
  - MemAvailable < 10% → 告警
  - Swap used > 20% → 告警
  - 进程OOM次数 → 告警

IO：
  - disk util > 80% → 告警
  - iops饱和 → 告警

进程：
  - zombie进程 > 0 → 告警
  - 文件描述符使用 > 80% → 告警
```
