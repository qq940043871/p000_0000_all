> 模块：操作系统
> 更新时间：2026-03-28

---

## 一、理论基础

### 1. 虚拟内存概念

```
物理内存 vs 虚拟内存：

物理内存：
  - 实际的RAM芯片容量
  - 所有进程共享
  - 访问地址固定

虚拟内存：
  - 每个进程独立的地址空间
  - 32位系统：4GB（用户空间3GB，内核1GB）
  - 64位系统：巨大的地址空间（实际受限于硬件）
  - 通过页表映射到物理内存

虚拟内存优势：
  - 进程隔离，互不影响
  - 简化内存管理
  - 支持内存交换（Swap）
  - 允许使用超过物理内存的地址空间
```

### 2. 分页机制

```
页（Page）：
  - 虚拟内存的基本单位
  - 常见大小：4KB（标准页）、2MB/1GB（大页）
  - 页框（Frame）：物理内存对应单位

页表结构：
  - 单级页表：简单但浪费空间（每个进程4MB）
  - 两级页表：节省空间，Linux使用
  - 三级/四级页表：64位系统使用

页表项内容：
  - 物理页框号
  - 访问权限（R/W/X）
  - 脏位（是否被修改）
  - 访问位（是否被访问）
  - PTE（Page Table Entry）

地址转换：
  虚拟地址（32位）= 页目录索引(10) + 页表索引(10) + 页内偏移(12)
  
  虚拟地址（64位）= PML4(9) + PDPT(9) + PD(9) + PT(9) + offset(12)
```

### 3. TLB（Translation Lookaside Buffer）

```
作用：缓存虚拟地址到物理地址的映射

命中：
  虚拟地址 → TLB查找 → 命中 → 物理地址
  延迟：~1ns

未命中：
  虚拟地址 → TLB查找 → 未命中 → 页表查找 → 更新TLB → 物理地址
  延迟：额外~10-100ns

TLBmiss代价：
  - 需要访问内存中的页表
  - 多次内存访问
  - 大页面可以减少TLBmiss

优化策略：
  - 大页内存（HugePages）：2MB/1GB
  - 预取：提前加载可能需要的页表项
  - TLB shootdown：修改页表时刷新TLB
```

### 4. 内存分配算法

```
连续内存分配：
  - 首次适配（First Fit）：简单，容易产生外部碎片
  - 最佳适配（Best Fit）：最小空闲块，容易产生细小碎片
  - 最差适配（Worst Fit）：最大空闲块，减少碎片但分配慢

非连续内存分配：
  - 分页：固定大小，消除外部碎片，但有内部碎片
  - 分段：按逻辑划分，大小不一
  - 段页式：先分段，再分页

Buddy System（Linux伙伴系统）：
  - 将内存按2的幂次划分为块
  - 分配：找到最小满足的块
  - 合并：相邻的同大小块合并
  - 优点：快速分配，减少碎片
  - 缺点：内部碎片（最多50%）

Slab分配器（Linux内核）：
  - 面向对象，缓存常用对象
  - 减少内核内存碎片
  - kmalloc基于Slab
```

### 5. 页面置换算法

```
FIFO（先进先出）：
  - 最简单，可能产生Belady异常（更多页面反而更少缺页）

LRU（最近最少使用）：
  - 最理想，但实现成本高
  - 需要硬件支持访问时间戳

Clock算法（近似LRU）：
  - 环形链表 + 访问位
  - 命中清除访问位
  - 置换时扫描清除位
  - Linux使用

工作集模型：
  - 保持活跃页面在内存
  - 非活跃页面逐步换出
  - 动态调整内存分配

OOM Killer（Linux特有）：
  - 内存耗尽时的最后手段
  - 选择"badness"最高的进程杀死
  - 优先杀死大内存进程
```

## 二、实践应用

### 1. 内存监控

```bash
# 基础监控
free -h                      # 总览内存使用
cat /proc/meminfo            # 详细内存信息

# vmstat
vmstat 1                     # 每秒更新
vmstat -s                    # 详细统计

# 实时监控
top                         # 查看%MEM列
htop                        # 更友好

# 进程内存详情
pmap -x <pid>               # 详细内存映射
cat /proc/<pid>/smaps       # 每段内存详情
cat /proc/<pid>/status | grep -i vm    # VmPeak/VmRSS等
```

### 2. 内存分析

```bash
# smem：更准确的内存统计
smem -p                     # 按百分比排序
smem -u                     # 按用户统计
smem -r -P java             # 只看java进程

# 查看具体内存分布
cat /proc/meminfo | grep -E "MemTotal|MemFree|MemAvailable|Buffers|Cached"

# 内存指标解释：
# MemTotal: 总内存
# MemFree: 完全空闲
# MemAvailable: 可用内存（包括可回收缓存）
# Buffers: 块设备缓冲区
# Cached: 文件系统缓存
# -/+ buffers/cache: 实际使用/可用
```

### 3. 内存优化

```bash
# 系统层面
# /etc/sysctl.conf
vm.swappiness=10           # 降低swap倾向
vm.dirty_ratio=15          # 脏页比例
vm.dirty_background_ratio=5 # 后台脏页比例
vm.overcommit_memory=1      # 允许过量分配（容器环境）
vm.max_map_count=655360    # 最大内存映射数

# 应用层面
# Java
-Xmx4g -Xms4g              # 初始和最大堆
-XX:+UseG1GC               # G1垃圾收集器
-XX:MaxRAMPercentage=70    # 最大堆占比（容器环境）

# 查看OOM Killer配置
cat /proc/sys/vm/oom_dump_tasks
cat /proc/sys/vm/oom_score_adj   # -1000到1000，越高越优先被杀
```

### 4. HugePages配置

```bash
# 查看当前状态
cat /proc/meminfo | grep -i huge

# 临时配置
echo 1024 > /proc/sys/vm/nr_hugepages

# 永久配置
cat >> /etc/sysctl.conf << 'EOF'
vm.nr_hugepages = 1024
EOF

# 查看HugePages信息
cat /proc/meminfo | grep Huge
# AnonHugePages: 透明大页（THP）
# HugePages_Total: 分配的HugePages总数
# HugePages_Free: 空闲的HugePages
# HugePages_Rsvd: 预留但未分配
```

## 三、生产环境问题案例

### 案例1：内存持续增长不回落

**问题现象：**
Java服务内存从启动时的1GB逐渐增长到8GB（堆上限），然后Full GC，但只回收一点点。

**分析过程：**
```bash
# 1. 查看内存趋势
# 监控显示：内存持续增长，不回落

# 2. 生成堆快照
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 分析堆内容
jmap -histo <pid> | head -50
# 结果：
#  num #instances #bytes class name
#  1: 1234567 89012340 [C  (char数组，String内部)
#  2: 234567 56789012 [Ljava.lang.Object;  (Object数组)
#  3: 34567 3456789 java/util/HashMap$Node

# 4. MAT分析
# 查找GC Root到泄漏对象的路径
# 发现：大量HashMap持有对象引用
```

**根因分析：**
静态HashMap作为缓存，持续添加元素但从不清理。

**解决方案：**
```java
// 问题代码
private static Map<String, Object> cache = new HashMap<>();

// 方案1：使用带容量限制的缓存
LoadingCache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(Duration.ofHours(1))
    .build(k -> loadFromDB(k));

// 方案2：WeakHashMap（适合临时缓存）
private static Map<String, WeakReference<Object>> cache = new WeakHashMap<>();

// 方案3：定期清理
ScheduledExecutorService cleaner = Executors.newSingleThreadScheduledExecutor();
cleaner.scheduleAtFixedRate(() -> {
    synchronized(cache) {
        cache.clear();
    }
}, 1, 1, TimeUnit.HOURS);
```

**经验教训：**
- 静态集合是内存泄漏首选
- 监控Old Gen使用率
- 使用成熟的缓存库而非手写

### 案例2：容器内存超限被Kill

**问题现象：**
Docker容器被OOMKilled，进程退出。

**分析过程：**
```bash
# 1. 查看容器日志
docker logs <container_id>
# 结果：Killed

# 2. 查看dmesg
dmesg | tail | grep -i memory
# [timestamp] Memory cgroup out of memory: Killed process

# 3. 查看容器配置
docker inspect <container_id> | grep -i memory
# "Memory": "4g"
# "MemorySwap": "4g"
# 发现：内存限制4GB，但JVM设置了8GB堆

# 4. 分析
# 4GB limit < 8GB heap + Metaspace + DirectMemory + Native
```

**根因分析：**
容器内存限制小于JVM堆内存 + 其他内存开销。

**解决方案：**
```yaml
# docker-compose.yml
services:
  app:
    image: java-app
    deploy:
      resources:
        limits:
          memory: 8G        # 至少等于JVM堆+2G
        reservations:
          memory: 4G
    environment:
      # JVM自动检测容器限制
      JAVA_OPTS: "-XX:+UseContainerSupport -XX:MaxRAMPercentage=70.0"
```

```bash
# Kubernetes
resources:
  limits:
    memory: "8Gi"
  requests:
    memory: "4Gi"
```

**经验教训：**
- 容器内存限制 = JVM堆 + Metaspace + DirectMemory + Native
- 推荐配置：容器limit = JVM堆×1.2
- 使用-XX:+UseContainerSupport让JVM自动适配

### 案例3：内存碎片导致可用内存远小于物理内存

**问题现象：**
服务器有128GB内存，但运行几十个进程后，无法分配10GB连续内存。

**分析过程：**
```bash
# 1. 查看内存
free -h
# Mem: total=128G, used=95G, free=33G

# 2. 查看buddyinfo（内存碎片）
cat /proc/buddyinfo
# Node 0, zone    DMA      freepages by size (order)
# Node 0, zone  Normal    1024 512 256 128 64 32 16 8 4 2 1
# 分析：高阶空闲块少，说明碎片化严重

# 3. 查看slab占用
cat /proc/meminfo | grep -i slab
# SReclaimable: 20GB（可回收）
```

**根因分析：**
长期运行后，物理内存碎片化，无法分配大块连续内存。

**解决方案：**
1. 临时：重启释放内存
2. 优化：使用大页内存减少碎片
   ```bash
   echo 32 > /proc/sys/vm/nr_hugepages  # 32×2MB = 64GB大页
   ```
3. 监控：关注buddyinfo高阶块数量
4. 预防：定期维护，避免长期不重启

**经验教训：**
- 内存碎片问题在大内存服务器上更明显
- 大页内存可以减少碎片
- 长期运行的服务器需要定期维护
