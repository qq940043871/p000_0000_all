## 一、理论基础

### 1. 进程 vs 线程

```
进程（Process）：
  - 资源分配的基本单位
  - 拥有独立的地址空间（堆、代码段、数据段）
  - 进程间通信需要IPC机制
  - 创建/切换开销大（毫秒级）
  - 隔离性好，一个崩溃不影响其他

线程（Thread）：
  - CPU调度的基本单位
  - 共享进程的地址空间（堆、全局变量）
  - 线程间通信直接读写共享数据
  - 创建/切换开销小（微秒级）
  - 隔离性差，一个崩溃可能影响整个进程

对比：
                进程              线程
-------------------------------------------------
创建速度        慢（10-100ms）   快（1-5ms）
切换速度        慢（1-10ms）     快（0.1-1ms）
内存占用        大（MB级）        小（KB级）
通信            复杂（IPC）       简单（共享内存）
稳定性          高                低
同步复杂度      低                高
```

### 2. 进程生命周期

```
                    ┌─────────┐
                    │   创建   │ fork()/vfork()
                    └────┬────┘
                         │
                         ▼
                    ┌─────────┐
              ┌─────│  就绪态  │◄────────────────┐
              │     └────┬────┘                   │
              │          │ 调度器选中              │ 时间片用完/高优先级
              │          ▼                        │ 进程恢复
              │     ┌─────────┐                   │
              │     │ 运行态  │───────────────────┘
              │     └────┬────┘
              │          │ 阻塞/等待I/O
              │          ▼
              │     ┌─────────┐
              └────►│ 阻塞态  │
                    └────┬────┘
                         │ I/O完成
                         ▼
                    ┌─────────┐
                    │  终止    │ exit()/信号
                    └─────────┘
```

### 3. 进程调度算法

```
批处理系统：
  - FCFS（先来先服务）：简单，但短作业可能等很久
  - SJF（最短作业优先）：最优平均等待时间，但难以预测执行时间
  - HRRN（最高响应比优先）：响应比 = (等待时间+执行时间)/执行时间

交互式系统（分时）：
  - RR（时间片轮转）：公平，但响应时间依赖时间片大小
  - 优先级调度：重要进程优先，但可能饿死
  - 多级反馈队列：综合最优，Linux/Unix使用

实时系统：
  - RM（速率单调）：周期越短优先级越高
  - EDF（最早截止时间优先）：截止时间越早优先级越高

Linux调度器：
  - O(1)调度器（2.6内核之前）：调度复杂度固定，但不适合大负载
  - CFS（完全公平调度器，2.6.23+）：
    * 红黑树管理就绪队列
    * 虚拟运行时间（vruntime）
    * nice值影响权重
    * 睡眠公平性支持
```

### 4. 上下文切换

```
上下文切换过程：
  1. 保存当前进程上下文（寄存器、PC、栈指针）
  2. 更新当前进程PCB（进程控制块）
  3. 调度器选择下一个进程
  4. 恢复下一个进程上下文
  5. 切换到新进程执行

上下文组成：
  - 通用寄存器
  - 程序计数器（PC）
  - 程序状态字（PSW）
  - 栈指针（SP）
  - 内存管理信息（页表基址）

成本：
  - 每次切换：2-5微秒（现代CPU）
  - 高频切换（如1秒10000次）：20-50毫秒CPU时间
  - TLB刷新：额外成本
```

### 5. 线程模型

```
用户线程（User Thread）：
  - 内核不知道线程存在
  - 线程切换在用户空间完成
  - 优点：快，不依赖内核
  - 缺点：一个线程阻塞，整个进程阻塞

内核线程（Kernel Thread）：
  - 内核直接管理线程
  - 线程切换需要内核参与
  - 优点：一个阻塞不影响其他
  - 缺点：创建/切换慢

混合模型（N:M）：
  - N个用户线程映射到M个内核线程
  - 兼顾效率和灵活性
  - Linux的线程实现（CLONE_SETTLS）

协程（Coroutine）：
  - 用户态的"轻量级线程"
  - 主动让出CPU（非抢占）
  - Go goroutine、Python asyncio、Lua协程
  - 百万级并发成为可能
```

### 6. 进程间通信（IPC）

```
管道（Pipe）：
  - 半双工，匿名管道用于父子进程
  - 命名管道（FIFO）可用于无亲缘关系进程
  - 示例：ls | grep txt

消息队列：
  - 消息链表，存放在内核中
  - 可按类型读取
  - 示例：System V msgget/msgsnd

共享内存：
  - 最快IPC方式（无需内核拷贝）
  - 需要信号量配合同步
  - 示例：shmget/shmat

信号量：
  - 计数器，用于进程/线程同步
  - P操作（减），V操作（加）
  - 示例：sem_wait/sem_post

信号：
  - 异步通知机制
  - 示例：SIGKILL、SIGSEGV、SIGALRM
  - kill -9 pid

Socket：
  - 网络通信，可用于本地（AF_UNIX）
  - 最通用，跨机器通信
```

## 二、实践应用

### 1. 进程管理命令

```bash
# 查看进程
ps aux                    # 所有进程
ps -ef                    # 详细格式
ps -T -p <pid>            # 查看线程
pstree -p                 # 进程树

# 实时监控
top                       # 交互式进程监控
htop                      # 更友好的top
btop                      # 现代美化版

# 进程操作
kill <pid>               # 发送TERM信号
kill -9 <pid>            # 强制杀死
kill -19 <pid>           # 暂停（SIGSTOP）
kill -18 <pid>           # 继续（SIGCONT）

# pgrep/pkill
pgrep -u username java    # 查找用户java进程
pkill -9 java             # 杀死所有java进程

# nice/renice
nice -n 10 ./program     # 启动时设置优先级
renice -n 5 -p <pid>     # 调整运行中进程优先级
```

### 2. 线程分析

```bash
# 查看进程线程数
ps -eLf | grep <process_name> | wc -l

# 查看线程ID
ps -T -p <pid>

# /proc查看
ls /proc/<pid>/task       # 线程ID列表
cat /proc/<pid>/status    # 进程状态
cat /proc/<pid>/wchan     # 阻塞位置

# pstack查看线程堆栈
pstack <pid>

# gdb查看线程
gdb -p <pid>
  (gdb) info threads
  (gdb) thread apply all bt
```

### 3. CPU亲和性

```bash
# taskset设置亲和性
taskset -c 0,2,4,6 java -jar app.jar   # 绑定到特定CPU
taskset -cp 0-7 <pid>                  # 查看当前亲和性

# isolcpus（启动参数）
# /etc/default/grub
GRUB_CMDLINE_LINUX="isolcpus=0,1,2,3"
# 隔离CPU，专用于特定任务

# cpuset（容器环境）
cpuset.cpus.effective /cpuset/xxx
```

### 4. 进程资源限制

```bash
# 查看限制
ulimit -a                  # 查看当前shell限制
cat /proc/<pid>/limits     # 查看进程限制

# 临时修改
ulimit -n 65535            # 文件描述符
ulimit -u 4096             # 最大进程数

# 永久修改
# /etc/security/limits.conf
* soft nofile 65535
* hard nofile 65535
* soft nproc 65535
* hard nproc 65535
```

## 三、生产环境问题案例

### 案例1：Java进程突然消失

**问题现象：**
应用进程莫名消失，没有任何日志。

**分析过程：**
```bash
# 1. 查看系统日志
dmesg | tail -100 | grep -i java

# 结果：
# [12345.678901] java[12345]: segfault at xxx ip xxx sp xxx error 4
# [12345.678902] oom-kill: idx=5 ... java ...

# 2. 查看OOM Killer
dmesg | grep -i "out of memory"
# 或
grep -i "killed process" /var/log/messages

# 3. 查看内存
free -h
vmstat 1
```

**根因分析：**
Java进程被OOM Killer杀死（out of memory killer）。

**解决方案：**
1. 增加物理内存
2. 降低Java堆内存（-Xmx）
3. 优化应用内存使用
4. 添加监控告警
```bash
# /etc/sysctl.conf
vm.overcommit_memory = 1  # 允许过量分配
```

**经验教训：**
- dmesg是排查进程异常消失的第一站
- Java堆不要超过物理内存70%
- 监控可用内存，提前告警

### 案例2：高并发下请求堆积

**问题现象：**
业务高峰期，请求响应时间暴增。

**分析过程：**
```bash
# 1. 查看系统负载
uptime
# load average: 15.32, 12.21, 10.15

# 2. 查看CPU和进程状态
top
# 发现：很多进程处于D状态（不可中断睡眠）

# 3. 查看IO状态
iostat -xz 1
# 发现：磁盘IO使用率100%

# 4. 查看进程IO
iotop
# 发现：备份进程正在大量写入
```

**根因分析：**
IO瓶颈导致进程阻塞，请求堆积。

**解决方案：**
1. 错峰执行备份
2. 使用ionice降低IO优先级
3. 独立磁盘用于备份
4. 增加缓存减少IO

**经验教训：**
- Load高不一定是CPU问题
- D状态进程通常是IO或网络等待
- 备份任务要使用低优先级

### 案例3：线程池耗尽

**问题现象：**
服务正常启动，但请求全部超时。

**分析过程：**
```bash
# 1. 查看进程状态
ps -eLf | grep java | wc -l
# 结果：200+

# 2. 查看线程堆栈
jstack <pid> | grep "pool" -A 5

# 3. 查看GC
jstat -gcutil <pid> 1000

# 发现：
# - 线程数达到上限（200）
# - 线程全部处于BLOCKED状态
# - 等待数据库连接
```

**根因分析：**
数据库连接池配置过小 + 数据库连接泄漏。

**解决方案：**
```java
// 优化连接池配置
HikariCP:
  maximum-pool-size: 50   # 增加
  minimum-idle: 10
  connection-timeout: 30000
  idle-timeout: 600000

// 添加连接泄漏检测
leakDetectionThreshold: 60000  # 60秒未归还视为泄漏
```

**经验教训：**
- 监控线程池活跃线程数
- 线程BLOCKED通常是等待外部资源
- 连接池要设置合理的超时和泄漏检测
