## 一、理论基础

### 1. 常见文件系统

```
Linux文件系统：
  ext4（第四代扩展文件系统）
    - 主流默认，稳定
    - 支持最大16TB文件，最大文件系统16EB
    - 日志功能（ext3/ext4）
    - 延迟分配，减少碎片

  XFS
    - 高性能大文件
    - 最大8EB文件，64位设计
    - 元数据日志，快速恢复
    - 适合数据库、视频存储

  btrfs（B-tree文件系统）
    - 写时复制（COW）
    - 快照、压缩、校验
    - 仍在发展中，生产慎用

  ZFS（Solaris移植）
    - 128位文件系统
    - 存储池概念
    - 内置RAID、数据校验
    - 内存需求高

Windows文件系统：
  NTFS
    - 默认Windows文件系统
    - 日志、权限、加密
    - 动态磁盘

  exFAT
    - 大文件支持（>4GB）
    - U盘常用

macOS文件系统：
  APFS（Apple File System）
    - SSD优化
    - 快照、加密
    - High Sierra+默认

  HFS+（旧版）
```

### 2. 文件系统结构

```
ext4结构：

Boot Sector     ┌─────────────────────────┐
                │ Boot Record / Boot Block │
                ├─────────────────────────┤
                │    Super Block          │  文件系统元数据
                │  (备份在Block Group)    │
                ├─────────────────────────┤
Block Group 0   │ Group Descriptor Table   │
                │ Block Bitmap            │  块分配位图
                │ Inode Bitmap            │  inode分配位图
                │ Inode Table             │  inode表（文件元数据）
                │ Data Blocks             │  实际数据
                ├─────────────────────────┤
Block Group 1   │ ...                     │
                └─────────────────────────┘

关键概念：
  Block（数据块）：最小存储单元，通常4KB
  Inode：文件的元数据（权限、时间、大小、块指针）
  Directory Entry：文件名到inode的映射
```

### 3. VFS（虚拟文件系统）

```
作用：统一抽象不同文件系统

         ┌────────────────────────────┐
         │        User Space          │
         ├────────────────────────────┤
         │        VFS Layer           │
         │  (read/write/open/close)   │
         ├────────────────────────────┤
         │  ext4 │ XFS │ btrfs │ NTFS │  ← 具体文件系统实现
         ├────────────────────────────┤
         │     Block Device Layer     │
         ├────────────────────────────┤
         │    Physical Device (Disk)  │
         └────────────────────────────┘

VFS提供统一的：
  - 文件描述符
  - inode结构
  - dentry（目录项）缓存
  - Page Cache（页缓存）
```

### 4. IO模型

```
同步阻塞IO：
  - 进程调用recvfrom，等待数据就绪后返回
  - 期间进程阻塞
  - 最简单，但效率最低

同步非阻塞IO：
  - 轮询检查数据是否就绪
  - 期间进程不阻塞，但CPU空转
  - 适合等待时间短的场景

IO多路复用：
  - select/poll：线性扫描fd集合，有数量限制
  - epoll：事件驱动，无fd数量限制，Linux特有
    * 水平触发（LT）：未处理继续通知
    * 边缘触发（ET）：状态变化才通知
  - 原理：红黑树 + 双向链表

异步IO（Linux AIO/libaio）：
  - 发起请求后立即返回
  - 数据就绪后通知进程
  - 实现复杂，应用较少

异步IO（Linux io_uring，5.1+）：
  - 两个环形缓冲区（submission/completion）
  - 零系统调用开销
  - 高性能IO场景新选择
```

### 5. Page Cache（页缓存）

```
原理：
  - 读取文件时，数据先加载到页缓存
  - 写入文件时，先写页缓存，后异步刷盘
  - 读取已缓存数据无需访问磁盘

关键参数：
  dirty_ratio：脏页占比超过此值，写进程阻塞
  dirty_background_ratio：后台刷盘阈值
  dirty_expire_centisecs：脏页过期时间
  dirty_writeback_centisecs：后台刷盘间隔

查看：
  cat /proc/meminfo | grep -E "Dirty|Writeback"
  vmstat 1

刷盘策略：
  - pdflush/flush：后台刷盘线程
  - sync：强制同步
  - fsync：单个文件强制刷盘
```

## 二、实践应用

### 1. 文件系统管理

```bash
# 查看文件系统
df -h

# 查看 inode
df -i

# 查看挂载选项
mount

# 查看具体文件占用
du -sh /var/log/*
du -h --max-depth=1 /

# 查看大文件
find / -type f -size +100M -exec ls -lh {} \;

# 修复文件系统（umount后）
fsck -y /dev/sda1
xfs_repair /dev/sda1
```

### 2. IO模型选择

```bash
# 查看当前IO调度器
cat /sys/block/sda/queue/scheduler
# [mq-deadline] none

# SSD建议用noop
echo "noop" > /sys/block/sda/queue/scheduler

# 数据库建议用deadline
echo "deadline" > /sys/block/sda/queue/scheduler

# 查看队列深度
cat /sys/block/sda/queue/nr_requests
```

### 3. epoll使用示例

```c
// epoll服务端示例
int epfd = epoll_create1(0);
struct epoll_event ev, events[MAX_EVENTS];

// 添加监听
ev.events = EPOLLIN;
ev.data.fd = listenfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listenfd, &ev);

while (1) {
    int nfds = epoll_wait(epfd, events, MAX_EVENTS, -1);
    for (int i = 0; i < nfds; i++) {
        if (events[i].data.fd == listenfd) {
            // 新连接
            int connfd = accept(listenfd, ...);
            ev.events = EPOLLIN | EPOLLET;  // 边缘触发
            epoll_ctl(epfd, EPOLL_CTL_ADD, connfd, &ev);
        } else {
            // 处理数据（边缘触发要一次读完）
            char buf[4096];
            while (read(events[i].data.fd, buf, sizeof(buf)) > 0) {
                process(buf);
            }
        }
    }
}
```

### 4. IO优化配置

```bash
# /etc/sysctl.conf

# 磁盘预读
blockdev --setra 4096 /dev/sda  # 预读4KB的倍数（8KB=2048）

# IO队列深度
echo 2048 > /sys/block/sda/queue/nr_requests

# 提交周期
echo 100 > /proc/sys/vm/dirty_writeback_centisecs

# SSD特殊优化
# 1. TRIM支持
fstrim -av  # 定期执行

# 2. 禁用atime（减少写操作）
mount -o noatime,nodiratime /dev/sda1 /mnt

# 3. 裸设备（数据库直接访问块设备，跳过文件系统）
```

## 三、生产环境问题案例

### 案例1：小文件过多导致inode耗尽

**问题现象：**
磁盘还有空间，但无法创建文件。

**分析过程：**
```bash
# 1. 查看文件系统
df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1       100G   95G  5G   95% /data

# 2. 查看inode
df -i
Filesystem       Inodes   IUsed  IFree  IUse% Mounted on
/dev/sda1      32768000 32768000       0 100% /data

# 3. 查找小文件
find /data -type f | wc -l
# 大量小文件

# 4. 统计目录
for dir in /data/*/; do
    echo -n "$dir: "
    find "$dir" -type f | wc -l
done
```

**根因分析：**
大量小文件（每个文件占用1个inode），inode耗尽。

**解决方案：**
1. 短期：删除无用文件
2. 迁移数据到新分区（inode充足）
3. 合并小文件（如打包、数据库存储）
4. 创建时预估inode需求

```bash
# 创建高inode文件系统
mkfs.ext4 -N 65536000 /dev/sdb1  # 指定inode数量
```

**经验教训：**
- ext4默认inode比例可能不够
- 大量小文件场景要预留足够inode
- 监控inode使用率

### 案例2：Nginx请求卡住

**问题现象：**
Nginx响应卡住，部分请求超时。

**分析过程：**
```bash
# 1. 查看Nginx状态
curl http://127.0.0.1/nginx_status

# 2. 查看连接状态
netstat -an | awk '/^tcp/ {s[$NF]++} END {for(k in s) print k, s[k]}'

# 3. 查看worker进程状态
ps -p <worker_pid> -L -o pid,tid,psr,pcpu,wchan
# 大量进程处于D状态（IO等待）

# 4. 查看IO
iostat -xz 1
# 发现磁盘100%繁忙

# 5. 分析
# 静态文件存储磁盘繁忙，请求排队
```

**根因分析：**
静态文件存储在机械硬盘，高并发下IO成为瓶颈。

**解决方案：**
1. 升级到SSD
2. 使用内存文件系统缓存热点文件
   ```bash
   mount -t tmpfs tmpfs /var/cache/nginx -o size=2G
   ```
3. 增加worker进程（但IO瓶颈不变）
4. 使用CDN分流静态请求

**经验教训：**
- IO密集型服务要考虑磁盘性能
- Nginx静态文件对IO敏感
- SSD vs HDD在高并发场景差距巨大

### 案例3：文件删除后磁盘空间不释放

**问题现象：**
删除了大文件，但df显示空间没变。

**分析过程：**
```bash
# 1. 查看磁盘使用
df -h /data
# 发现：used=80G，available=20G

# 2. 查找大文件（已删除但未关闭）
lsof | grep deleted | sort -k7 -nr | head -10
# 结果：mysqld 某个数据文件，deleted，大小50GB

# 3. 分析
# 文件被删除，但进程仍在使用（持有文件描述符）
# 需要重启进程或关闭文件
```

**根因分析：**
进程持有已删除文件的文件描述符，空间无法释放。

**解决方案：**
```bash
# 方法1：重启进程
systemctl restart mysqld

# 方法2：截断文件（不推荐，可能导致进程异常）
# > /proc/<pid>/fd/<fd_number>

# 方法3：找到进程关闭文件
lsof +L1  # 查看无链接文件
kill -9 <pid>
```

**经验教训：**
- lsof +L1是排查此类问题的神器
- 日志文件要配置轮转（logrotate）
- 删除文件前确保无进程使用
