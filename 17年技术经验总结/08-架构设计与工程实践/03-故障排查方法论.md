## 一、排查流程

### 1. 标准流程

```
发现问题
  ↓
确认影响范围
  ↓
快速止血（临时方案）
  ↓
定位根因
  ↓
制定修复方案
  ↓
实施修复
  ↓
复盘总结
```

### 2. 信息收集

```bash
# 系统信息
uptime
top
vmstat 1
iostat -xz 1

# 网络信息
netstat -s
ss -s
tcpdump -i eth0

# 应用信息
jstack <pid> > thread.log
jmap -dump:format=b,file=heap.hprof <pid>
jstat -gcutil <pid> 1000

# 日志
tail -f /var/log/app.log
grep -i error /var/log/*.log
```

## 二、常见问题排查

### 1. CPU飙高

```bash
# 1. 找到高CPU进程
top -H -p <pid>

# 2. 转换线程ID
printf "%x\n" <tid>

# 3. 查看线程堆栈
jstack <pid> | grep <hex-tid> -A 50

# 4. 分析热点
perf top -p <pid>
```

### 2. 内存泄漏

```bash
# 1. 查看GC
jstat -gcutil <pid> 1000

# 2. 生成堆快照
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 分析堆
# 使用MAT或VisualVM

# 4. 查看大对象
jmap -histo <pid> | head -30
```

### 3. 响应慢

```bash
# 1. 查看系统负载
vmstat 1

# 2. 查看IO
iostat -xz 1

# 3. 查看网络
netstat -an | awk '/^tcp/ {s[$6]++} END {for(k in s) print k, s[k]}'

# 4. 查看线程状态
jstack <pid> | grep -E "BLOCKED|WAITING"

# 5. 查看数据库慢查询
# MySQL: show processlist
# Redis: slowlog get 10
```

## 四、案例分析

### 案例：支付超时

**现象：**
支付请求超时，用户无法完成支付。

**排查过程：**
```
1. 查看监控
   - 支付接口响应时间：5000ms（正常<200ms）
   - 数据库连接池：满载

2. 查看数据库
   show processlist;
   → 发现大量慢查询

3. 分析慢查询
   EXPLAIN SELECT * FROM orders WHERE status=1;
   → 全表扫描

4. 修复索引
   ALTER TABLE orders ADD INDEX idx_status (status);

5. 验证效果
   → 响应时间恢复到200ms
```

**复盘：**
- 索引缺失导致全表扫描
- 高并发放大问题
- 缺少索引监控
