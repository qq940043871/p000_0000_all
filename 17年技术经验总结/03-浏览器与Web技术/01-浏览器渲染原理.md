## 一、理论基础

### 1. 浏览器组成

```
Chrome/Chromium架构：

┌─────────────────────────────────────┐
│           Browser Process           │  浏览器主进程
│  (UI、地址栏、书签、前进后退)        │
├─────────────────────────────────────┤
│         Renderer Process(es)        │  渲染进程
│  (每个标签页一个，沙箱隔离)          │
│  ├── HTML Parser                     │
│  ├── CSS Parser                      │
│  ├── JavaScript Engine (V8)         │
│  ├── Layout Engine (Blink)           │
│  └── Compositor                      │
├─────────────────────────────────────┤
│            GPU Process               │  GPU进程
│  (图形渲染、合成)                    │
├─────────────────────────────────────┤
│            Network Process           │  网络进程
│  (网络请求、DNS、缓存)               │
├─────────────────────────────────────┤
│           Plugin Process            │  插件进程
│  (Flash、PDF等)                      │
└─────────────────────────────────────┘

多进程优势：
  - 隔离崩溃：一个标签页崩溃不影响其他
  - 安全沙箱：渲染进程无法直接访问系统
  - 充分利用多核：各进程并行
```

### 2. 渲染流水线

```
渲染流水线（Rendering Pipeline）：

HTML ──► DOM Tree ──► Render Tree ──► Layout ──► Paint ──► Composite

1. HTML解析 → DOM Tree
   bytes → characters → tokens → nodes → DOM Tree

2. CSS解析 → CSSOM Tree
   bytes → characters → tokens → nodes → CSSOM Tree

3. DOM + CSSOM → Render Tree
   - 只包含可见节点（display:none不包含）
   - 计算样式

4. Layout（布局/排版）
   - 计算每个节点的位置和大小
   - 回流（Reflow）：几何属性改变

5. Paint（绘制）
   - 分层绘制指令
   - 重绘（Repaint）：外观改变，位置不变

6. Composite（合成）
   - 图层合并
   - GPU加速
   - 最终显示
```

### 3. 重绘与回流

```
重绘（Repaint）：
  - 触发条件：外观改变（color、background-color、visibility）
  - 代价：较低，不需要重新布局
  - 优化：使用transform/opacity（开启GPU加速）

回流（Reflow/Layout）：
  - 触发条件：几何属性改变
    * width、height、margin、padding
    * display、position、top、left
    * font-size、font-family
    * offsetWidth、offsetHeight（读取布局属性）
  - 代价：非常高，需要重新计算布局
  - 注意：回流必定触发重绘

批量优化：
  // 错误：多次回流
  element.style.width = '100px';
  element.style.height = '100px';
  console.log(element.offsetWidth); // 强制回流

  // 正确：批量修改
  element.style.cssText = 'width:100px;height:100px;';
  
  // 最佳：使用transform
  element.style.transform = 'translateX(100px)'; // 不触发回流
```

### 4. JavaScript执行

```
JavaScript引擎（以V8为例）：

源代码 ──► 解析器 ──► AST ──► 解释器(Ignition) ──► 字节码
                              │
                              └──► 编译器(Turbofan) ──► 机器码
                                   (热点代码优化编译)

关键概念：
  - JIT编译：热点代码即时编译
  - Hidden Class：对象形状优化
  - 内联缓存：属性访问优化
  - 垃圾回收：分代回收（New/Old Space）

执行上下文：
  - 全局上下文：window/global对象
  - 函数上下文：每次函数调用创建
  - Eval上下文：eval代码执行

作用域链：
  当前作用域 → 外部作用域 → ... → 全局作用域

事件循环：
  1. 执行同步任务
  2. 微任务队列（Promise.then、MutationObserver）
  3. 宏任务队列（setTimeout、setInterval、I/O、UI渲染）
  4. 重复
```

### 2. 渲染性能优化

```javascript
// 1. 减少回流
// 批量DOM操作
const fragment = document.createDocumentFragment();
items.forEach(item => {
  const li = document.createElement('li');
  li.textContent = item;
  fragment.appendChild(li);
});
container.appendChild(fragment);

// 2. 使用CSS类批量修改
element.classList.add('active', 'large');

// 3. 缓存布局属性
const width = element.offsetWidth; // 读取一次
requestAnimationFrame(() => {
  element.style.width = width + 10 + 'px'; // 使用缓存值
});

// 4. 使用transform/opacity
// 开启图层，触发Composite，不触发Layout/Paint
element.style.transform = 'translateX(100px)'; // GPU加速
element.style.opacity = 0.5; // GPU加速

// 5. will-change提示
element.style.willChange = 'transform';
// 完成后移除
element.style.willChange = 'auto';
```

### 3. 长任务优化

```javascript
// 1. 使用requestIdleCallback
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && tasks.length > 0) {
    processTask(tasks.shift());
  }
});

// 2. 使用Web Worker
const worker = new Worker('worker.js');
worker.postMessage(data);
worker.onmessage = (e) => {
  console.log('Result:', e.data);
};

// 3. 任务分片
function processInChunks(items, process, chunkSize = 100) {
  let index = 0;
  function doChunk() {
    const chunk = items.slice(index, index + chunkSize);
    chunk.forEach(process);
    index += chunkSize;
    if (index < items.length) {
      setTimeout(doChunk, 0);
    }
  }
  doChunk();
}
```

### 4. 内存优化

```javascript
// 1. 避免内存泄漏
// 常见泄漏场景：
// - 全局变量
// - 闭包
// - 事件监听未移除
// - setInterval未清除
// - 缓存无限制增长

// 2. 正确清理事件监听
class Component {
  constructor() {
    this.handler = this.onClick.bind(this);
    element.addEventListener('click', this.handler);
  }
  destroy() {
    element.removeEventListener('click', this.handler);
    // 清理引用
    this.handler = null;
  }
}

// 3. 使用WeakMap/WeakSet
const cache = new WeakMap(); // 自动回收
// 4. 懒卸载大型数据结构
```

## 三、生产环境问题案例

### 案例1：页面加载白屏

**问题现象：**
页面加载时长时间白屏，用户体验差。

**分析过程：**
```javascript
// 1. 检查网络请求
// DevTools Network面板
//   - 阻塞资源？CSS/JS加载顺序？
//   - 大文件下载？

// 2. 检查渲染阻塞
//   - JS是否阻塞HTML解析？
//   - defer/async属性使用

// 3. 检查关键渲染路径
//   - 首屏内容是否在首屏HTML中？
//   - 图片是否延迟加载？
```

**解决方案：**
```html
<!-- 1. CSS放在head中，JS放在body末尾 -->
<link rel="stylesheet" href="styles.css">
<!-- defer：HTML解析完成后执行 -->
<script src="analytics.js" defer></script>
<!-- async：下载完成立即执行 -->
<script src="ads.js" async></script>

<!-- 2. 内联关键CSS -->
<style>/* 关键渲染路径CSS */</style>

<!-- 3. 预加载关键资源 -->
<link rel="preload" href="font.woff2" as="font" crossorigin>
<link rel="preconnect" href="https://api.example.com">

<!-- 4. 图片优化 -->
<img loading="lazy" src="image.jpg" alt="">
```

**经验教训：**
- 关键CSS内联，首屏不等待
- JS异步加载，不阻塞渲染
- 图片懒加载

### 案例2：滚动卡顿

**问题现象：**
页面滚动时明显卡顿，不流畅。

**分析过程：**
```javascript
// 1. 检查帧率
// DevTools Performance，查看FPS线是否平稳

// 2. 检查长任务
// 滚动时是否有JS执行超过16ms

// 3. 检查重绘/回流
// 滚动事件处理是否触发布局？

// 4. 检查合成层
// 元素是否触发额外图层？
```

**根因分析：**
滚动事件处理中执行了复杂计算或DOM操作。

**解决方案：**
```javascript
// 1. 使用passive事件监听
window.addEventListener('scroll', handler, { passive: true });

// 2. 节流/防抖滚动处理
function throttle(func, limit) {
  let inThrottle;
  return function(...args) {
    if (!inThrottle) {
      func.apply(this, args);
      inThrottle = true;
      setTimeout(() => inThrottle = false, limit);
    }
  };
}

// 3. 使用CSS will-change
.scrolling-element {
  will-change: transform;
}

// 4. 内容虚拟化（长列表）
// 使用react-window或vue-virtual-scroller
```

**经验教训：**
- 滚动处理要轻量
- 使用passive优化
- 内容虚拟化处理长列表

### 案例3：内存持续增长

**问题现象：**
单页应用运行时间越长，内存占用越高，浏览器变卡。

**分析过程：**
```javascript
// 1. 录制堆快照
// Memory面板 → Take heap snapshot
// 对比不同时间的快照

// 2. 分配时间线
// Memory面板 → Allocation Timeline
// 观察内存增长点

// 3. 常见泄漏源
//   - 事件监听未移除
//   - 全局变量
//   - 闭包引用
//   - 定时器未清理
//   - DOM引用
```

**根因分析：**
组件销毁时未清理事件监听器和定时器。

**解决方案：**
```javascript
class Modal {
  constructor() {
    this.onClose = this.handleClose.bind(this);
    this.handleEsc = (e) => e.key === 'Escape' && this.close();
    document.addEventListener('keydown', this.handleEsc);
  }
  
  handleClose() {
    // 清理所有监听
    document.removeEventListener('keydown', this.handleEsc);
    this.close();
  }
  
  destroy() {
    this.handleClose();
    // 清理定时器
    clearTimeout(this.timeoutId);
    clearInterval(this.intervalId);
  }
}

// React/Vue使用useEffect清理
useEffect(() => {
  window.addEventListener('resize', handler);
  return () => window.removeEventListener('resize', handler);
}, []);
```

**经验教训：**
- 组件生命周期要完整清理
- 使用工具检测内存泄漏
- 定期测试内存增长
