# 1.2 设计目标与原则

## 一、核心设计目标

### 1. 高可靠性（Reliability）

消息是业务数据的载体，任何一条消息的丢失都可能造成业务损失。

| 目标 | 指标 | 实现手段 |
|------|------|----------|
| 消息不丢失 | 持久化消息落盘成功率 ≥ 99.999% | 同步刷盘 + 副本机制 |
| 消息不重复 | 重复消息率 < 0.001% | 幂等设计 + 去重索引 |
| 消息不乱序 | 严格有序队列乱序率 = 0 | 单分区 + 序列号机制 |

### 2. 高性能（Performance）

| 目标 | 指标 | 说明 |
|------|------|------|
| 高吞吐 | 单节点 ≥ 10万 条/秒 | 批量写入 + 零拷贝 |
| 低延迟 | P99 延迟 < 10ms | 内存队列 + 异步IO |
| 高并发 | 支持 ≥ 10000 并发连接 | IO多路复用 + 连接池 |

### 3. 高可用（Availability）

| 目标 | 指标 | 实现方式 |
|------|------|----------|
| 故障自愈 | 自动故障检测 < 5s | 心跳检测 + 租约机制 |
| 集群容错 | 节点故障不影响服务 | 主从复制 + 自动切换 |
| 滚动升级 | 升级过程零停机 | 蓝绿部署 + 流量迁移 |

### 4. 可扩展性（Scalability）

- **水平扩展**：通过增加节点来扩展集群容量
- **分区扩展**：支持按队列维度分片存储
- **协议扩展**：插件化协议接入层

### 5. 可观测性（Observability）

- 全链路 tracing：消息从生产到消费全程可追溯
- 指标暴露：Prometheus / OpenTelemetry 兼容
- 统一日志：结构化日志，支持 ELK 查询

## 二、核心设计原则

### 原则 1：分层解耦

```
协议层 ──▶ 路由层 ──▶ 队列层 ──▶ 存储层
```

各层职责单一，通过定义良好的接口通信。协议层可插拔，存储层可替换。

### 原则 2：最终一致性优先

在分布式环境下，强一致性会带来高昂的协调成本。本系统采用**最终一致性**模型，允许在极端故障情况下短暂的数据不一致，但保证最终一定会达到一致状态。

### 原则 3：幂等设计贯穿全链路

- 生产者：消息发送支持幂等 ID
- 存储层：消息写入去重
- 消费者：业务处理幂等化

### 原则 4：配置与数据分离

- **元数据**（队列定义、用户权限）存储在独立元数据库
- **业务数据**（消息体、索引）存储在消息存储引擎
- 分离后元数据变更无需移动消息数据

### 原则 5：向后兼容

- 数据库表结构变更支持在线 DDL
- 协议字段增加采用宽容解析（忽略未知字段）
- API 版本化管理，支持平滑升级

## 三、反模式与约束

| 反模式 | 说明 | 正确做法 |
|--------|------|----------|
| 单点写入 | 所有消息写入同一节点 | 队列分区 + 负载均衡 |
| 无限队列 | 队列消息无限堆积 | 设置最大长度 + 溢出策略 |
| 同步刷盘阻塞 | 每条消息同步刷盘导致吞吐下降 | 批量刷盘 + 页缓存 |
| 缺乏超时 | 消息消费无超时控制 | 设置消费超时 + 死信处理 |
| 硬编码协议 | 紧耦合特定协议 | 协议抽象层 + 插件化 |
