# 10.3 分区分表策略

## 概述

当单机存储成为瓶颈时，需要通过数据分片（Sharding）将数据和负载分散到多个节点。本章节提供完整的分片策略设计与实施指南。

## 分片键选择策略

### 分片键应满足

| 原则 | 说明 |
|------|------|
| 均匀分布 | 能将数据均匀分散到各分片 |
| 高查询友好 | 常见查询尽量在单分片完成 |
| 递增性 | 避免热点分片（追加写优于随机写）|
| 业务相关 | 尽量贴合业务逻辑 |

### 常见分片键

| 分片键 | 适用场景 | 缺点 |
|--------|---------|------|
| `user_id % N` | 用户维度的队列（私信、通知）| 跨用户查询需聚合 |
| `tenant_id % N` | 多租户场景 | 租户数据倾斜 |
| `timestamp` | 时间序列队列 | 热点分片（当前数据）|
| `message_id % N` | 无特定规律 | 无法范围查询 |
| `region_id % N` | 地域分布场景 | 跨地域查询需聚合 |

## 分片算法

### 1. 哈希分片（Hash Sharding）

```java
// 算法
shard_id = hash(sharding_key) % num_shards

// 示例：按 user_id 分片
shard_id = Math.abs(user_id.hashCode()) % 4

// 分片映射
shard_0: user_id % 4 == 0
shard_1: user_id % 4 == 1
shard_2: user_id % 4 == 2
shard_3: user_id % 4 == 3
```

**优点**：数据分布均匀
**缺点**：扩缩容时数据需要大规模迁移

### 2. 一致性哈希（Consistent Hashing）

```java
// 一致性哈希环（2^160 个节点）
// 虚拟节点：每个物理节点映射 150 个虚拟节点

// 添加节点：只需迁移 1/N 的数据
// 删除节点：只需迁移 1/N 的数据

// Ketama 算法
shard_id = findNodeOnRing(hash(key), ring)
```

### 3. 范围分片（Range Sharding）

```sql
-- 按时间分片
CREATE TABLE message_partitioned (
    message_id VARCHAR(64),
    created_at DATETIME,
    payload BLOB,
    PRIMARY KEY (message_id, created_at)
) PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
    PARTITION p202601 VALUES LESS THAN (202602),
    PARTITION p202602 VALUES LESS THAN (202603),
    PARTITION p202603 VALUES LESS THAN (202604),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

## 数据库分片设计

### 表结构：shard_config（分片配置表）

```sql
CREATE TABLE shard_config (
    shard_id           INT               NOT NULL COMMENT '分片ID',
    shard_name         VARCHAR(64)       NOT NULL COMMENT '分片名称',
    
    -- 节点配置
    primary_node       VARCHAR(128)      NOT NULL COMMENT '主节点',
    replica_nodes      JSON              DEFAULT NULL COMMENT '副本节点列表',
    /*
    ["broker-replica-01", "broker-replica-02"]
    */
    
    -- 分片规则
    sharding_type      ENUM('hash', 'range', 'directory') NOT NULL,
    sharding_key       VARCHAR(64)       DEFAULT NULL COMMENT '分片键',
    sharding_algorithm  VARCHAR(256)     DEFAULT NULL COMMENT '分片算法表达式',
    
    -- 范围（range 类型）
    range_start        VARCHAR(255)      DEFAULT NULL COMMENT '范围起始值',
    range_end          VARCHAR(255)      DEFAULT NULL COMMENT '范围结束值',
    
    -- 容量
    max_capacity       BIGINT            DEFAULT 100000000 COMMENT '最大容量(条)',
    current_count      BIGINT            NOT NULL DEFAULT 0 COMMENT '当前消息数',
    current_size_bytes BIGINT            NOT NULL DEFAULT 0 COMMENT '当前存储(字节)',
    
    -- 状态
    status             ENUM('active', 'readonly', 'draining', 'offline') DEFAULT 'active',
    weight             INT               DEFAULT 100 COMMENT '负载权重(1-1000)',
    
    -- 位置
    datacenter         VARCHAR(64)       DEFAULT NULL COMMENT '所属机房',
    rack               VARCHAR(64)       DEFAULT NULL COMMENT '所属机架',
    
    -- 审计
    created_at         DATETIME(3)       NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    updated_at         DATETIME(3)       NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    
    PRIMARY KEY (shard_id),
    KEY uk_name (shard_name)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci
  COMMENT = '分片配置表';
```

### 表结构：shard_rebalance_job（分片重平衡任务表）

```sql
CREATE TABLE shard_rebalance_job (
    job_id             VARCHAR(64)       NOT NULL COMMENT '任务ID',
    job_type           ENUM('move', 'split', 'merge', 'rebalance') NOT NULL,
    
    -- 源和目标
    source_shard_id    INT               NOT NULL COMMENT '源分片ID',
    target_shard_id    INT               DEFAULT NULL COMMENT '目标分片ID',
    
    -- 范围
    move_key_start     VARCHAR(255)      DEFAULT NULL COMMENT '迁移键起始',
    move_key_end       VARCHAR(255)      DEFAULT NULL COMMENT '迁移键结束',
    
    -- 进度
    total_records      BIGINT            NOT NULL DEFAULT 0 COMMENT '总记录数',
    moved_records      BIGINT            NOT NULL DEFAULT 0 COMMENT '已迁移数',
    progress_pct       DECIMAL(5,2)     NOT NULL DEFAULT 0 COMMENT '进度百分比',
    
    -- 限速
    rate_limit_records INT               DEFAULT 10000 COMMENT '迁移速率限制(条/秒)',
    pause_on_high_load BOOLEAN          NOT NULL DEFAULT TRUE COMMENT '高负载时暂停',
    
    -- 状态
    status             ENUM('pending', 'running', 'paused', 'completed', 'failed', 'cancelled') DEFAULT 'pending',
    started_at         DATETIME(3)       DEFAULT NULL,
    completed_at       DATETIME(3)       DEFAULT NULL,
    
    -- 错误
    error_message      TEXT              DEFAULT NULL,
    retry_count        INT               NOT NULL DEFAULT 0,
    
    PRIMARY KEY (job_id),
    INDEX idx_status (status),
    INDEX idx_source (source_shard_id)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4 COLLATE = utf8mb4_unicode_ci
  COMMENT = '分片重平衡任务表';
```

## 分片后跨分片查询

### 广播表（所有分片冗余）

```sql
-- 配置表，所有分片都保留一份
-- 用于小表、低频更新的配置数据

CREATE TABLE sharded_config (
    config_key VARCHAR(128),
    config_value TEXT,
    updated_at DATETIME(3),
    PRIMARY KEY (config_key)
) ALL SHARDS;
```

### 抑或表（ES/搜索）

```java
// 不能按分片键查询的消息，路由到 ES
public List<Message> searchByContent(String keyword, String vhost) {
    // 1. 查询 ES 获取匹配的消息 ID
    List<String> messageIds = esClient.searchIds("content:" + keyword);
    
    // 2. 按 message_id 路由到各分片查询
    Map<Integer, List<String>> grouped = groupByShard(messageIds);
    
    // 3. 聚合结果
    List<Message> results = new ArrayList<>();
    for (Map.Entry<Integer, List<String>> entry : grouped.entrySet()) {
        results.addAll(shardClient.get(entry.getKey()).getMessages(entry.getValue()));
    }
    return results;
}
```

## 分片扩缩容流程

### 扩容（新加分片）

```
步骤：
1. 新增分片节点和配置
2. 启用新分片（状态=active）
3. 启动增量数据迁移
   $ openclaw shard rebalance start --new-shard 5 --rate-limit 5000
4. 验证数据完整性
   $ openclaw shard check integrity --shard 5
5. 调整路由配置
6. 下线旧分片（如需缩容）
```

### 缩容

```
步骤：
1. 标记目标分片为 draining
2. 等待数据迁移完成
3. 验证所有消息已迁移
4. 从路由表移除
5. 删除分片数据
6. 释放节点资源
```
