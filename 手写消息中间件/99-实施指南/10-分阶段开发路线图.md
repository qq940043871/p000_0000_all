# 分阶段开发路线图

> 本文档将消息中间件的开发分解为 4 个阶段，每个阶段有明确的目标、交付物和里程碑。

---

## 路线图总览

```
Phase 1 (4-6周) ──▶ Phase 2 (6-8周) ──▶ Phase 3 (4-6周) ──▶ Phase 4 (4-6周)
  单机核心            协议+高可用          高级特性           运维+生态
```

---

## Phase 1：单机核心引擎（第 1-6 周）

### 目标
实现单机版消息中间件核心功能，支持基本的消息收发。

### 交付物

| 序号 | 功能 | 优先级 | 交付时间 |
|------|------|--------|---------|
| 1 | 数据库初始化脚本 | P0 | Week 1 |
| 2 | 连接管理与心跳 | P0 | Week 1-2 |
| 3 | AMQP 协议解析 | P0 | Week 2-3 |
| 4 | Direct/Fanout/Topic 交换机 | P0 | Week 2-3 |
| 5 | Classic Queue 实现 | P0 | Week 3-4 |
| 6 | 消息持久化存储 | P0 | Week 4-5 |
| 7 | 消费者 Pull 模式 | P0 | Week 4-5 |
| 8 | Basic ACK/NACK 确认 | P0 | Week 5-6 |
| 9 | Management REST API | P1 | Week 6 |

### 里程碑：MVP (Minimum Viable Product)
```
Week 6 验收标准：
✅ 支持 AMQP 客户端连接
✅ 创建/删除队列
✅ 发送消息到队列
✅ 消费者拉取并确认消息
✅ 消息持久化（重启不丢）
✅ Management API 查看队列状态
```

### 技术任务分解

```
Week 1: 基础设施
├── 数据库初始化（建表SQL）
├── 项目脚手架
├── 配置管理
└── 日志框架

Week 2-3: 协议层
├── Netty 框架搭建
├── AMQP 帧编解码
├── Connection/Channel 管理
└── 心跳检测

Week 3-4: 路由层
├── Exchange 抽象
├── Direct/Fanout/Topic 实现
└── Binding 解析

Week 4-5: 存储层
├── Commit Log 实现
├── 消息追加写
├── 消息读取
└── 消息删除

Week 5-6: 消费层
├── Consumer 注册
├── Pull 模式实现
├── ACK/NACK 处理
└── 重投机制
```

## 风险识别与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Raft 实现复杂度高 | 高 | Phase 1 先实现单机，Phase 2 引入 etcd 或 SOFA JRaft 库 |
| 高并发下的性能瓶颈 | 中 | 提前做性能测试，Phase 1 就引入 JMH 压测 |
| MQTT 协议兼容性问题 | 中 | 使用成熟协议库（EMQX 项目中的库）|
| 存储引擎选型错误 | 高 | Phase 1 末期做存储压测对比（Kafka/RocksDB/LevelDB）|

## 验收测试用例

### Phase 1 验收用例

```
[ ] 连接管理
    [ ] 正常连接可建立
    [ ] 心跳超时自动断开
    [ ] 非法用户拒绝连接

[ ] 消息发送
    [ ] 持久消息重启后存在
    [ ] 非持久消息重启后丢失
    [ ] 消息属性（TTL/Priority）生效

[ ] 消息消费
    [ ] Pull 模式正常拉取
    [ ] ACK 后消息删除
    [ ] NACK 后消息重投

[ ] 交换机路由
    [ ] Direct 精确匹配
    [ ] Fanout 广播
    [ ] Topic 通配符匹配

[ ] Management API
    [ ] 列出所有队列
    [ ] 查看队列消息数
    [ ] 查看消费者数
```
