# 系统负载高排查案例

> 模块：实战案例
> 更新时间：2026-03-29

---

## 问题描述

```
服务器Load Average: 50 (16核CPU)
CPU使用率只有30%
用户抱怨应用响应缓慢
```

---

## 问题分析

```
Load高但CPU低的原因:
1. I/O等待 (最常见)
2. 进程在不可中断睡眠(D状态)
3. 锁竞争
```

---

## 排查过程

### 第一步：查看整体状态

```bash
# 查看负载
uptime
# 10:30:00 up 30 days, 5:42, 2 users, load average: 50.23, 48.15, 45.32

# 查看CPU和进程
top
# Cpu(s): 30.2%us, 5.1%sy, 0.0%ni, 64.0%id, 0.7%wa
# 发现: wa(等待)较高

# vmstat分析
vmstat 1 10
# procs -----------memory---------- -swap- -----io---- -system- --------cpu--------
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs  us  sy  id  wa  st
# 52  8      0  2048M   100M    10G    0    0  5000  2000   0    0  30   5   64   7   0

# 发现: b列(阻塞进程)=8, wa=7%
```

### 第二步：找出阻塞进程

```bash
# 查看D状态进程(不可中断睡眠)
ps aux | awk '$8 ~ /D/ {print}'

# 输出:
# USER  PID %CPU %MEM  VSZ  RSS S   TIME COMMAND
# root 1234  0.0  0.1 100M  50M D    0:00 [java]
# root 1235  0.0  0.1 100M  50M D    0:00 [java]

# 查看这些进程的堆栈
cat /proc/1234/wchan
# 输出: blk_mq_get_tag

# 说明在等待磁盘I/O
```

### 第三步：分析I/O

```bash
# iostat分析
iostat -x 1 5

# 输出:
# Device  rrqm/s  wrqm/s  r/s   w/s  rkB/s  wkB/s avgrq-sz avgqu-sz await r_await w_await svctm %util
# sda       0.00   100.00  0.00 200.00 0.00  800.00   8.00    40.00 200.00  0.00 200.00 5.00 100.00

# 发现: sda磁盘使用率100%,await=200ms(很慢)

# iotop找出I/O大户
iotop -o

# 输出:
# Total DISK READ:         0.00 B/s | Total DISK WRITE:       2.00 M/s
#   TID  PRIO  USER     DISK READ  DISK WRITE  SWAPIN  IO>    COMMAND
#  12345 be/4 root        0.00 B/s    2.00 M/s    --  99.00%   java
```

### 第四步：定位Java应用

```bash
# 查看Java进程
ps aux | grep java | head -5

# 查看打开的文件
lsof -p 12345 | head -20

# 发现大量日志文件
# java   12345 root   123w   REG    8,0 1073741824  12345678 app.log

# 日志文件1GB+,正在持续写入

# 查看文件写入情况
cat /proc/12345/fd/123 | xxd | head
```

### 第五步：日志分析

```bash
# 查看日志内容
tail -f /var/log/app.log

# 发现大量ERROR日志
# 每秒写入1000+条

# 找到日志来源
grep -n "ERROR" /var/log/app.log | head -10
```

---

## 根因分析

```
1. 应用产生大量ERROR日志
2. 日志写入导致大量磁盘I/O
3. 磁盘是普通HDD,性能不足
4. 进程等待I/O进入D状态
5. Load升高,但CPU不高
```

---

## 解决方案

### 临时方案

```bash
# 1. 日志轮转
logrotate -f /etc/logrotate.d/app

# 2. 清理旧日志
> /var/log/app.log  # 清空日志(保留文件)

# 3. 临时降级服务
systemctl restart app
```

### 永久方案

```bash
# 1. 配置logrotate
cat > /etc/logrotate.d/app << 'EOF'
/var/log/app*.log {
    daily
    rotate 7
    missingok
    notifempty
    compress
    delaycompress
    copytruncate
    maxsize 100M
}
EOF

# 2. 使用异步日志
# logback配置
<appender name="ASYNC" class="ch.qos.logback.classic.AsyncAppender">
    <queueSize>512</queueSize>
    <discardingThreshold>0</discardingThreshold>
    <includeCallerData>false</includeCallerData>
    <appender-ref ref="FILE"/>
</appender>

# 3. 限制日志级别
# logback配置
<logger name="com.example" level="WARN"/>
```

## 总结

```
排查流程:
1. uptime看Load
2. top看CPU和wa(等待)
3. ps看D状态进程
4. iostat看磁盘
5. iotop找I/O大户
6. lsof + fd找文件

关键命令:
- uptime
- top
- vmstat
- ps aux | awk '$8 ~ /D/'
- iostat -x
- iotop
- lsof -p PID
```
