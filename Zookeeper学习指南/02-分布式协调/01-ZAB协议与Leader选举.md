# ZAB协议与Leader选举

> 模块：分布式协调
> 更新时间：2026-04-06

---

## 一、ZAB协议概述

### 1.1 ZAB协议定义

```
ZAB (ZooKeeper Atomic Broadcast) 协议

是ZooKeeper专门设计的一种支持崩溃恢复的原子广播协议

核心功能:
1. 原子广播 - 保证消息广播的原子性
2. 崩溃恢复 - Leader故障后的恢复机制
```

### 1.2 协议模式

```
ZAB协议两种模式:

┌─────────────────────────────────────────────────────────────┐
│                      ZAB协议                                │
│  ┌─────────────────────┐  ┌─────────────────────┐          │
│  │   消息广播模式       │  │   崩溃恢复模式       │          │
│  │  (Broadcast Mode)   │  │  (Recovery Mode)    │          │
│  │                     │  │                     │          │
│  │  - 正常工作状态     │  │  - Leader选举       │          │
│  │  - 处理客户端请求   │  │  - 数据同步         │          │
│  │  - 原子广播消息     │  │  - 恢复一致性       │          │
│  └─────────────────────┘  └─────────────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

## 二、消息广播

### 2.1 广播流程

```
消息广播流程:

┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Client  │────▶│  Leader  │────▶│ Follower │────▶│ Follower │
└──────────┘     └────┬─────┘     └────┬─────┘     └────┬─────┘
                      │                │                │
                      ▼                ▼                ▼
                 ┌─────────────────────────────────────────┐
                 │              2PC 两阶段提交              │
                 │  1. Leader发起Proposal                   │
                 │  2. Follower返回ACK                      │
                 │  3. Leader收到过半ACK后Commit            │
                 └─────────────────────────────────────────┘
```

### 2.2 两阶段提交

```
阶段1: Proposal
1. Leader收到写请求
2. 生成全局唯一ZXID
3. 向所有Follower发送Proposal

阶段2: Commit
1. 等待过半Follower的ACK
2. 向所有Follower发送Commit
3. Follower执行Commit

过半机制:
- 收到超过半数ACK即可Commit
- 不需要等待所有Follower响应
```

### 2.3 ZXID结构

```
ZXID (64位):
┌─────────────────────────────────┬─────────────────────────┐
│         高32位                   │        低32位            │
│      epoch (纪元)               │   counter (计数器)       │
│      Leader任期                 │   消息序号               │
└─────────────────────────────────┴─────────────────────────┘

示例:
ZXID = 0x0000000100000001
- epoch = 1 (第1任Leader)
- counter = 1 (第1条消息)
```

## 三、Leader选举

### 3.1 选举触发条件

```
触发Leader选举的情况:
1. 集群初始化启动
2. Leader网络中断
3. Leader崩溃重启
4. Leader与过半Follower断开
```

### 3.2 选举流程

```
Leader选举流程:

阶段1: 发现阶段
┌─────────────────────────────────────────────────────────────┐
│  每个Server向其他Server发送选举投票                         │
│  投票内容: (SID, ZXID)                                      │
│  SID: 服务器ID                                              │
│  ZXID: 最后处理的事务ID                                     │
└─────────────────────────────────────────────────────────────┘

阶段2: 领导发现
┌─────────────────────────────────────────────────────────────┐
│  1. 收到其他Server的投票                                    │
│  2. 比较ZXID，大的优先                                      │
│  3. ZXID相同比较SID，大的优先                               │
│  4. 获得过半投票的Server成为Leader                          │
└─────────────────────────────────────────────────────────────┘

阶段3: 数据同步
┌─────────────────────────────────────────────────────────────┐
│  1. 新Leader建立新的epoch                                   │
│  2. Follower向Leader同步数据                                │
│  3. 确保所有Server数据一致                                  │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 选举算法

```
投票比较规则:

投票1: (sid1, zxid1)
投票2: (sid2, zxid2)

比较逻辑:
if (zxid1 > zxid2) {
    投票1胜出
} else if (zxid1 < zxid2) {
    投票2胜出
} else {
    // ZXID相同，比较SID
    if (sid1 > sid2) {
        投票1胜出
    } else {
        投票2胜出
    }
}

选举结果:
- 获得超过半数投票的Server成为Leader
- 其他Server成为Follower
```

## 四、崩溃恢复

### 4.1 恢复场景

```
场景1: Leader崩溃
1. Follower检测到Leader心跳超时
2. 触发Leader选举
3. 选出新Leader
4. 数据同步

场景2: Follower崩溃
1. Leader检测到Follower心跳超时
2. 将Follower从ISR移除
3. Follower恢复后同步数据

场景3: 网络分区
1. 少数派分区无法与Leader通信
2. 多数派分区选举新Leader
3. 分区恢复后，少数派加入新集群
```

### 4.2 数据一致性保证

```
ZAB协议保证:

1. 全局有序
   - 所有消息按ZXID顺序处理
   - 每个消息有唯一ZXID

2. 因果有序
   - 有因果关系的消息按顺序处理
   - 使用ZXID保证顺序

3. 崩溃恢复一致性
   - 新Leader确保所有已提交消息被同步
   - 丢弃未提交的消息
```

### 4.3 恢复流程

```
崩溃恢复流程:

┌─────────────────────────────────────────────────────────────┐
│ 1. 检测故障                                                 │
│    - 心跳超时                                               │
│    - 网络中断                                               │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 触发选举                                                 │
│    - 进入LOOKING状态                                        │
│    - 发起投票                                               │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ 3. 选举Leader                                               │
│    - 比较投票                                               │
│    - 过半确认                                               │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 数据同步                                                 │
│    - 确定最新数据                                           │
│    - 同步到所有Follower                                     │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│ 5. 恢复服务                                                 │
│    - 进入BROADCAST模式                                      │
│    - 开始处理请求                                           │
└─────────────────────────────────────────────────────────────┘
```

## 五、服务器状态

### 5.1 状态定义

```
ZooKeeper服务器状态:

LOOKING    - 选举状态，正在寻找Leader
FOLLOWING  - 跟随者状态，同步Leader数据
LEADING    - 领导者状态，处理写请求
OBSERVING  - 观察者状态，只同步数据不参与投票
```

### 5.2 状态转换

```
状态转换图:

                    ┌─────────────┐
                    │   LOOKING   │
                    │  (选举中)   │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               │               ▼
    ┌─────────────┐        │        ┌─────────────┐
    │  FOLLOWING  │        │        │   LEADING   │
    │  (跟随者)   │        │        │  (领导者)   │
    └──────┬──────┘        │        └──────┬──────┘
           │               │               │
           └───────────────┴───────────────┘
                           │
                    Leader故障时
                    回到LOOKING
```

## 六、配置参数

### 6.1 选举相关配置

```properties
tickTime=2000
initLimit=10
syncLimit=5

electionAlg=3

leaderServes=yes

cnxTimeout=5000
```

### 6.2 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| tickTime | 3000 | 心跳时间(ms) |
| initLimit | 10 | 初始同步超时(tickTime倍数) |
| syncLimit | 5 | 同步超时(tickTime倍数) |
| electionAlg | 3 | 选举算法 |
| leaderServes | yes | Leader是否处理客户端连接 |
